## 1. 文書の位置付け

本書は OriDesign v1 の唯一の規範的な要件ベースラインである。v1.0 や他の既存文書と矛盾する場合は本書を優先する。ADR は決定理由と変更履歴であり、要件本文へ反映されるまでは単独で本書を上書きしない。技術設計、画面仕様、開発計画および受け入れテストは本書に追従しなければならない。

機能実装の前に、後述の M0F（折り可能性保証の実現性ゲート）を完了する。M0F で保証契約を満たせないと判明した場合、保証を暗黙に弱めたり「既知の制限」として実装を続行したりしてはならない。要件変更 ADR を作成し、プロダクトオーナーの再承認を得る。

本書の「必須」は v1 リリース条件、「努力目標」は未達でも保証契約を弱めず、測定結果と改善課題を公開する条件を表す。

## 3. v1 の保証契約

### 3.1 用語

曖昧な「折れる」「妥当」を避けるため、次の用語を使用する。

- **読込可能入力**: スキーマと安全性検査を通過し、エディタまたはビューアで開けるデータ。保証対象とは限らない。
- **保証対象入力**: 設計プロジェクトでは、葉が 2～20 枚であり、連結・閉路なし、枝長が正、枝幅が 0 以上であるツリーのうち、M0F で確定・公開する生成用対応プロファイル（次数、形状、幅、数値範囲等）を満たす入力。FOLD プロジェクトでは、別の FOLD 検証用対応プロファイル（頂点・辺・面、割当て、縮退条件等）を満たす入力。
- **検証済み**: 本節の厚さゼロ数値モデルと公開された許容誤差の下で、平面の用紙から目的状態までの連続折り経路、面の非貫通、および積層順序を確認できた状態。
- **解なし**: 保証対象入力ではあるが、指定用紙、枝長、枝幅、方式および箱ひだ格子の対応範囲内に解が存在しないことを、M0F で定める判定根拠により確定した状態。
- **対応範囲外**: スキーマ上は読込可能でも、公開された対応プロファイルを満たさない状態。解なしとは区別する。
- **未検証**: 折り経路の検証が完了していない CP/候補の状態。インポート直後、手動 M/V 編集後、stale、または検証失敗を含む。ジョブのキャンセル自体は永続状態にせず、直前状態を維持する。キャンセル後に部分候補を明示採用する場合だけ、その候補を未検証とする。
- **対応プロファイル（`SupportProfile`）**: 方式ごとの保証対象入力を、ソルバー実行前に決定できる版付きの条件集合。
- **許容誤差プロファイル（`ToleranceProfile`）**: 距離、角度、頂点統合、接触、貫通、目標残差等の版付き数値集合。
- **revision vector**: 設計入力を表す `design`、配置を表す `layout`、CP 幾何と M/V を表す `cp` の三成分。成果物は、自身が依存する成分と生成元 hash に束縛する。名称、ラベル、表示設定はこれらの成分を進めない。
- **検証記録（`VerificationRecord`）**: 検証器版、各プロファイル、revision vector、入力・CP・経路の hash と検査結果を結ぶ証跡。
- **未検証候補（candidate）**: CP または経路を生成できたが、検証記録が成立していない下書き。

### 3.2 数値モデル上の「折れる」

v1 が保証する「折れる」とは、次をすべて満たすことである。

1. 紙は厚さゼロで、面は剛体、変形は定義された折り線上で起きる。
2. 設計プロジェクトでは、初期の平面状態から入力ツリーの枝構造・長さ比・幅を反映した平坦な uniaxial base まで、連続な折り経路が存在する。FOLD プロジェクトでは、現在の CP 幾何と M/V に整合する完全平坦目標までの連続経路が存在する。
3. 折り途中を含め、異なる面の接触と最終状態での重なりは許可するが、面同士の貫通は許可しない。
4. 接触・重なりには矛盾のない上下の積層順序を持たせる。
5. 距離、角度、接触、衝突の各許容誤差を満たす。許容誤差は紙の短辺を 1 として正規化し、検証結果に実際の値を表示する。
6. 前川・川崎の局所条件だけで保証済みとはしない。連続衝突判定を含む経路全体の検証を必須とする。

これは規定した数値モデル上の保証であり、数学的な形式証明、有限の厚み・素材の曲げ・摩擦を含む実物の紙、または人間が手順どおりに折れることの保証ではない。

100% は数学上の完全な平坦状態を表す。3D 描画では数値安定性のため、その直前の状態を表示してよいが、画面にその差を明示する。

### 3.3 保証される終端状態

- キャンセルされない限り、すべての保証対象入力は **検証済み** または **解なし** のどちらかへ確定しなければならない。
- 検証済み結果には、CP、整合した面情報、連続折り経路、衝突検証結果、積層順序、使用した許容誤差、対象 revision vector および生成元 hash を保存する。
- 解なしには、失敗箇所、制約、方式固有の探索範囲、および再現可能な判定根拠を表示・保存する。
- 単なるタイムアウト、探索打切り、数値発散またはキャンセルを「解なし」と表示してはならない。
- 固定の強制タイムアウトは設けない。処理は進捗を表示し、いつでもキャンセル可能とする。
- キャンセル時は途中結果で既存データを上書きせず、最後の検証済み結果を維持する。
- M0F で本契約を葉 20 枚まで満たせない場合、v1 の機能実装へ進まず、要件を再審議する。葉 20 枚を一方的に試験機能へ降格してはならない。

### 3.4 未検証データ

- 未検証または検証失敗した CP も、明瞭な警告付きで表示、M/V 編集、3D ビューへの読込、再検証、エクスポートを許可する。
- 未検証の候補アニメーションを表示する場合、検証済み折り経路と同じ外観・名称を使用してはならない。
- 「検証済み」の表示は、折り経路が依存する現在の revision vector 成分、入力 hash、各プロファイル版および検証器版が検証記録と一致する場合に限る。

## 4. 機能要件

### FR-1 アプリケーションとプロジェクト

- **FR-1.1** 新規設計プロジェクトを作成し、名称、用紙縦横比、設計方式を指定できる。
- **FR-1.2** 用紙は内部で正規化座標として扱う。幅・高さは縦横比であり、mm 等の実寸型紙を意味しない。
- **FR-1.3** 設計プロジェクトと FOLD プロジェクトの各々に UUID を付与する。
- **FR-1.4** `.orid.json` を保存・読込できる。FOLD だけを読み込んだ場合は、ツリーを持たない「FOLD プロジェクト」として M/V 編集、折り経路、検証状態を保存できる。
- **FR-1.5** 同一ファイルへの上書きをブラウザがサポートする場合は上書き保存し、非対応時はダウンロードへフォールバックする。`Ctrl+S` は同じ処理を実行する。上書きAPIの成功、またはダウンロード後の利用者による保存確認が得られた場合だけ未保存状態を解除する。ダウンロード開始だけを成功扱いしない。
- **FR-1.6** `.orid.json` と `.fold` は実行時スキーマ検証を行い、破損、上限超過、重複 ID、NaN/無限大、不正座標、不正参照、配列長の不整合を理由付きで拒否する。サイズ・要素数上限は M0F で確定する。

### FR-2 ツリーエディタ

- **FR-2.1** ノードの追加、移動、削除、および枝の接続・削除をキャンバス上で行える。
- **FR-2.2** 各枝は一意 ID、ラベル、正の実数の長さ、および 0 以上の幅 `w` を持つ。幅は葉への末端枝だけでなく内部枝にも設定でき、完成した枝または river の横幅を表す。長さと同じ比率単位を使う。
- **FR-2.3** 枝属性を一覧表とキャンバスの双方から編集し、双方向に同期する。
- **FR-2.4** エディタ上の枝の巡回順を設計上の埋込み順序として確定し、画面位置だけに依存しない明示データとして保存する。
- **FR-2.5** 左右対称モードでは左右を実ノード・実枝として保持し、対応を `mirrorOf` 相当の参照で保存する。軸上要素は自己対称とする。対称モードを OFF にしても生成済みの左右両方を残す。
- **FR-2.6** 未接続、葉不足、閉路などの編集途中状態を許可する。その状態では生成操作を無効化し、修正理由を表示する。
- **FR-2.7** 連結性、閉路、葉数、長さ、幅、参照整合性および保証対象プロファイルを常時検査する。

### FR-3 配置と設計方式

- **FR-3.1** ツリー法は、明示保存された埋込み順序、枝長、内部枝を含む枝幅、用紙比率および対称制約を用いて葉配置を求める。
- **FR-3.2** 箱ひだは、単にツリー法の葉座標を格子へ丸める方式ではなく、本来の格子型箱ひだとして方式固有の配置・軸線・river・分子・必要な折り線を構成する。詳細な構成法と対応プロファイルは M0F で確定する。
- **FR-3.3** 長方形の箱ひだ格子は正方形セルを維持し、横分割数 `Nx` と縦分割数 `Ny` を別々に管理する。両方向のセルピッチは公開許容誤差内で等しくなければならない。`Nx=4` または `Ny=4` を含む小さい格子を一律に禁止しない。上限は M0F で決定する。
- **FR-3.4** 箱ひだは実数の枝長・幅を受け付け、格子値へ近似する。各値の相対誤差 1% 以内を合格とし、元の値、格子値、誤差を表示する。1% 以内にできない場合は、より細かい格子候補を提示するか、対応格子範囲内での解なしを返す。
- **FR-3.5** 分岐部に指定幅が収まらない等、用紙・長さ・幅の条件から解が存在しない場合は解なしとする。
- **FR-3.6** 自動配置に加えて手動微調整を許可し、変更後は関連結果を stale にする。
- **FR-3.7** 箱ひだとツリー法を切り替えるとき、ツリーは保持し、配置、CP、M/V 上書き、折り経路、検証結果を現在状態から除去する。箱ひだへの切替時は保証条件を満たす格子候補を提案する。方式切替自体は Undo 対象とする。

### FR-4 CP 生成と検証

- **FR-4.1** 各方式は保証対象入力から、整合した平面グラフとして CP を生成する。交点での線分分割、許容誤差内の頂点統合、辺の一意化、面列挙を行い、FOLD の `faces_vertices` を構築する。
- **FR-4.2** 再生成後も対応付け可能な安定した折り線 ID を持ち、M/V 上書きと診断を追跡できる。
- **FR-4.3** 前川定理と川崎定理を内部頂点ごとに検査する。前川違反は M/V 手動編集の対象にできる。川崎違反は M/V の変更では直らないことを明示し、生成失敗または未検証ドラフトとして扱う。
- **FR-4.4** M/V 自動割当てを行い、CP 上で手動変更できる。手動変更時は検証済み状態を解除し、再検証に成功するまで未検証とする。
- **FR-4.5** 折り経路検証では、離散フレームの点検だけでなく、フレーム間を含む連続衝突判定を行う。接触と貫通を区別し、積層順序の一貫性を検査する。
- **FR-4.6** 検証結果画面に、状態、対象/生成元 revision vector、許容誤差、違反箇所、衝突した面の組、経路上の時刻、積層順序、および解なし・対応範囲外の理由を表示する。
- **FR-4.7** 重い計算は UI をブロックせず、ジョブ ID、対象 revision vector、入力 hash、進捗、キャンセルを持つ。依存成分または hash が一致しない古いジョブの結果を新しい入力へ適用してはならない。

### FR-5 FOLD 読込

- **FR-5.1** FOLD 1.1/1.2 の top-level key frame にある 2D 平面 CP を正式対応する。`file_frames` は存在しないか空配列でなければならず、有効フレームは厳密に 1 件とする。
- **FR-5.2** `faces_vertices` がない場合、辺の交点分割と頂点統合を行って面を再構築する。
- **FR-5.3** 1 件以上の `file_frames` を持つ複数フレーム、初期状態が 3D、非平面、非多様体、assignment `C/J`、または面を一意に再構築できないデータは、非対応理由を表示する。
- **FR-5.4** 読込直後は未検証とする。CP 表示、M/V 編集、再出力、3D ビューへの読込を許可し、検証成功後だけ検証済みとする。
- **FR-5.5** `.fold` 内の custom verification metadata は参考情報として表示してよいが、検証済み状態の復元には使用しない。ローカル検証器が現在の CP と証跡を再検証した場合だけ検証済みとする。
- **FR-5.6** FOLD プロジェクトの検証目標は、現在の CP 幾何を変えず、M/V を尊重し、未割当てを明示解決した無矛盾な完全平坦状態とする。M0F で生成用とは別の FOLD 検証用 `SupportProfile` と停止根拠を確定する。範囲外でも CP 表示、M/V 編集、警告付き再出力、平面または未検証候補の 3D 表示を妨げない。

### FR-6 CP ビュー、3D、状態表示

- **FR-6.1** CP を山折り、谷折り、境界、フラット/未割当てで区別し、ズーム、パン、全体表示、箱ひだ格子表示および参照座標表示を提供する。
- **FR-6.2** 状態は色だけでなく、線種、アイコン、テキストでも区別する。
- **FR-6.3** 3D ビューは回転、ズーム、パン、面の表裏色分け、積層確認および衝突診断を提供する。
- **FR-6.4** 検証済み結果の 0～100% スライダーは、全折り線を同率で動かす値ではなく、検証済み折り経路上の時刻を表す。各折り線は経路に従い異なる速度・タイミングで動く。
- **FR-6.5** 未検証 CP でも 3D タブを開けるが、検証済み経路がない場合は静的表示または「未検証の候補」として明示した表示に限る。

### FR-7 自動復元、複数タブ、Undo/Redo

- **FR-7.1** 編集停止から 2 秒後に IndexedDB へ自動保存し、最近使った 10 プロジェクトを復元候補として保持する。
- **FR-7.2** 起動時に無断で復元せず、復元または破棄を利用者が選択する。ファイル保存成功後も復旧用データを保持し、利用者が個別に削除できる。
- **FR-7.3** 同一 UUID のプロジェクトを複数タブで開いた場合、後から開いたタブを読取り専用とし、「こちらで編集を引き継ぐ」操作を用意する。
- **FR-7.4** Undo/Redo は既定 100 件、利用者が変更可能、最大 500 件とする。ドラッグ 1 回を 1 操作として記録する。
- **FR-7.5** ツリー、用紙、方式、格子、対称、手動配置、M/V 編集およびソルバー結果の反映を履歴対象とする。ソルバー結果の反映は 1 操作とする。配色等の端末設定は対象外とする。
- **FR-7.6** 配色、履歴上限、3D 表示設定は端末共通設定とし、プロジェクトファイルには含めない。

### FR-8 revision と stale 管理

- **FR-8.1** `design`、`layout`、`cp` からなる revision vector と、配置・CP・折り経路・検証結果それぞれが依存する成分、生成元 hash およびプロファイル版を保存する。
- **FR-8.2** 長さ、幅、用紙、格子、対称、ツリー構造または配置の変更で、それらに依存する CP と検証結果を stale にする。方式変更は FR-3.7 の特則によりツリー以外を破棄し、`none` とする。M/V、CP 幾何、`SupportProfile`、`ToleranceProfile` または検証器版の変更は検証結果を stale にする。名称、ラベルまたは表示色だけの変更では stale にしない。
- **FR-8.3** stale な旧 CP は保持して表示できるが、「再生成が必要」と常時明示し、検証済みの表示を解除する。
- **FR-8.4** キャンセル、失敗、古い revision のジョブ完了によって、最後の検証済み結果を上書きしない。

### FR-9 エクスポート

- **FR-9.1** FOLD 1.2、SVG、PNG、PDF を出力する。FOLD 1.2 の custom property 規則に従い、標準 reader が拡張を無視しても CP を読めるようにする。
- **FR-9.2** PNG の既定は長辺 4,096 px、最大 16,384 px とし、白背景と透明背景を選択できる。
- **FR-9.3** PDF の既定は 1 ページ、余白 10 mm、縦横自動、用紙中央への最大フィットとする。これは指定用紙へのフィット出力であり、原寸型紙ではない。
- **FR-9.4** SVG と FOLD は正規化座標を保持する。
- **FR-9.5** 未検証または stale なデータも警告確認後に出力できる。出力に検証状態、revision vector、許容誤差を記録し、メタデータを確実に保持できない形式では可視の警告を付ける。

### FR-10 PWA と更新

- **FR-10.1** v1 をインストール可能な PWA とする。PWA インストール UI はブラウザが当該機能を提供する場合に必須とし、提供しない対応ブラウザでは公開 URL からの利用を正式な代替経路とする。完全オフライン動作はリリース条件にしない。
- **FR-10.2** 新バージョン検出時に作業中のアプリを自動更新しない。「新しいバージョンがあります。保存後に更新」と通知し、利用者の操作後に更新する。

## 5. 非機能要件

### NFR-1 対応環境と画面

- Windows 10/11 の最新版 Chrome、Edge、Firefox を必須対応とする。Safari は対象外とする。
- 1920×1080 を標準とし、画面幅 1,280 px 未満では利用警告を表示する。モバイル最適化は行わない。

### NFR-2 性能と応答性

- すべての重い配置、CP 生成、経路生成、衝突検証はバックグラウンド処理とし、進捗表示とキャンセルを提供する。
- 葉 20 枚の代表ケースで、箱ひだ配置 3 秒以内、ツリー法配置 30 秒以内を努力目標とする。保証判定を打ち切る強制タイムアウトには使わない。
- 3D 表示は 1,000 三角形で 30 fps 以上を必須とし、2,000 三角形を努力目標とする。折り経路の検証時間とは別に測定する。
- 性能基準端末は Windows 11、4 コア CPU、メモリ 16 GB、内蔵 GPU、1920×1080 を目安とし、CPU 型番、ブラウザ版、測定手順および代表ケースを M0F で固定する。

### NFR-3 数値品質と再現性

- 紙の短辺を 1 として正規化し、距離、角度、接触、衝突ごとに別の許容誤差を使用する。具体値、丸め方、決定境界および再現性条件は M0F で確定する。
- 同一バージョン、同一入力、同一設定では、検証状態と判定根拠が再現可能でなければならない。
- ソルバー、幾何、検証部のユニットテストカバレッジは 80% 以上とし、境界値テストとゴールデンテストを設ける。

### NFR-4 プライバシーと通信

- アプリ配信に必要な同一オリジンの静的アセット取得と PWA 更新確認を除き、実行時に外部通信を行わない。
- プロジェクト、CP、検証結果、利用状況を送信しない。アクセス解析、広告、クラッシュ自動送信、外部 API、外部フォント、CDN を使用しない。

### NFR-5 アクセシビリティ

- 通常 UI とフォームは WCAG 2.2 AA を目標とする。
- ツリーと CP の選択・編集をキーボードでも行えるようにする。
- 色だけに依存せず線種、アイコン、テキストを併用する。
- 3D ビューの完全な代替操作が困難な部分は、代替の数値/状態表示を可能な範囲で提供し、既知制限として公開する。

### NFR-6 信頼性と更新安全性

- 破損入力や計算失敗で最後の検証済み結果と復旧用データを失わない。
- PWA 更新、複数タブ、古い Worker ジョブ、ファイル保存失敗を原因とする無言のデータ上書きを防ぐ。
- ファイル上限は性能・メモリ・拒否時 UX を測定して M0F で確定する。

### NFR-7 ライセンスと公開

- ソースコードとアプリ URL を公開し、本体を MIT ライセンスとする。
- 初期許可 SPDX ID は `MIT`、`0BSD`、`BSD-2-Clause`、`BSD-3-Clause`、`Apache-2.0`、`ISC` とし、CI で直接・推移依存を自動検査する。追加は ADR、法的互換性確認、allowlist 更新を必須とする。
- GPL その他コピーレフトのコードを依存、コピー、翻案、参照移植しない。論文および公開された数理記述を根拠に独立実装する。

## 6. M0F: 折り可能性保証の実現性ゲート

M0F は一般 UI 実装より前に行う、捨ててもよい試作と検証である。次を一体として実証する。

1. 本来の格子型箱ひだの方式固有 CP 生成。
2. 葉 2～20 枚の公開対応プロファイル内での CP 生成、または正しい解なし判定。
3. 平面グラフの交点分割、頂点統合、面列挙および整合した FOLD 面の構築。
4. 設計プロジェクトでは平面から目的の平坦 uniaxial base、FOLD プロジェクトでは CP/M/V に整合する完全平坦目標までの連続折り経路生成。
5. 接触を許し貫通を禁止する連続自己衝突判定。
6. 経路全体で矛盾しない面の積層順序。
7. ツリーを持たない対応 FOLD に対する、完全平坦目標、経路、CCD、積層の検証と終了判定。

M0F の必須成果物は次のとおりとする。

- 方式別の生成用保証対象プロファイル: 許容次数、ツリー形状、幅・長さ・用紙比率の数値範囲、必要な前提。ただし葉 20 枚までという v1 上限は維持する。
- FOLD 検証用保証対象プロファイル: 頂点・辺・面数、assignment、縮退、目標状態および停止根拠。
- 箱ひだ構成規則、`Nx`/`Ny` の探索範囲と上限、および 1% 近似判定方法。
- 距離、角度、接触、衝突、頂点統合、完全平坦直前表示に使う各許容誤差。
- 検証済み、解なし、対応範囲外、未検証を決定するアルゴリズムと、解なし判定根拠の形式。
- `.orid.json`/`.fold` のバイト数、頂点数、辺数、面数等の入力上限。
- 性能基準 CPU 型番、ブラウザ版、測定手順、代表入力および実測値。
- M0F 試作コード、再実行可能なテスト、結果レポート、既知の数値境界。
- 保証契約を満たして v1 実装へ進めるかを明記した Go/No-Go 判定。

上記は「実装時に適当に決める値」ではない。M0F の結果を ADR と技術設計へ反映し、テストで固定してから後続マイルストーンへ進む。

## 7. v1 の検証例とリリース条件

最低限、次の固定例を自動テストまたは再現可能なベンチマークに含める。

- rabbit ear
- waterbomb
- 4 葉の鳥の基本形
- 8 葉の昆虫型
- 固定した 20 葉ツリー
- 幅付き内部枝を含む例
- 長方形用紙の本来の箱ひだ例

v1 は次をすべて満たすまで公開版としない。

1. M0F が Go であり、公開された対応プロファイル内の葉 20 枚までで保証契約を満たす。
2. M0～M9 相当の機能と PWA 化が完了している。
3. 上記固定例、解なし例、対応範囲外例、衝突例、ファイル破損例のテストが通る。
4. Windows 対応ブラウザ、アクセシビリティ、保存復元、複数タブ、更新安全性、エクスポートの受け入れテストが通る。
5. 対応範囲、数値保証の意味、未検証表示、実物の紙を保証しない点を利用者向け文書に明記する。

## 8. v1 スコープ外

- 人間向けの折り順図解・ステップバイステップ手順の自動生成
- 有限の紙厚、素材特性、曲げ、摩擦、破れ、手指の可達性を含む物理保証
- 数学的な形式証明書の生成
- 段折りによる立体化（旧 M10。後続リリース候補）
- 3D 曲面設計
- Tauri 等によるデスクトップアプリ配布（後続候補）
- Safari およびモバイル最適化
- 完全オフライン環境へのインストール・動作保証
- クラウド保存、アカウント、共同編集、作品共有、テレメトリ
- mm 単位の原寸型紙出力
- 複数フレーム、初期 3D、非平面、非多様体または assignment `C/J` FOLD の正式対応
