## 1. 文書の目的

本書は、OriDesign v1 の中核要件がブラウザ上で実現可能かを、製品コードの実装前に判定するための仕様である。この検証マイルストーンを **M0F（Feasibility Gate）** と呼ぶ。

M0F は、UI の試作や「見た目上折れた」デモを作る工程ではない。次の主張を、再現可能な試作、検証データおよび独立検証器によって裏付けられるかを判定する。

> 公開された対応入力範囲内では、葉数20以下のツリーから本来の箱ひだ方式またはツリー法による展開図と平坦なuniaxial baseへの連続経路を生成する。また、対応FOLDではCP/M/Vに整合する完全平坦目標への連続経路を生成する。いずれも厚さゼロの剛体面が自己貫通せず、規定の数値許容誤差内でのみ「検証済み」と表示する。

M0F が `GO` になるまでは、ソルバー、3D 折り経路、自己衝突検証を製品仕様どおり実装し始めてはならない。M0F 用コードは捨ててもよい探索用プロトタイプとし、製品 UI、状態管理、エクスポート機能の先行実装を M0F の成功証拠には数えない。

本書の「必須」「禁止」は M0F の合否条件を表す。

## 2. M0F で固定して扱う製品契約

| 項目 | 固定する内容 |
|---|---|
| 紙 | 切断・伸縮・面内変形をしない、厚さゼロの単一長方形。短辺を `1` に正規化する |
| 面 | 折り線間を剛体平面として扱う |
| ヒンジ | 直線折り線を共有する隣接面だけが、その直線を軸に相対回転する |
| 初期状態 | 全面が同一平面上にあり、折り角が `0` の展開状態 |
| 目標状態 | 設計入力はツリーの枝構造、長さ比、幅を反映した平坦なuniaxial base。FOLD入力はCP/M/Vに整合する完全平坦状態 |
| 折り経路 | 初期状態から目標状態まで連続し、全時刻で面の剛性とヒンジ拘束を満たす |
| 接触 | 面同士の接触、接線接触および最終状態での重なりを許可する |
| 禁止事項 | 非隣接面の横断的な交差、面が別の面を通り抜けること、矛盾した積層順序 |
| 積層 | 重なる面の上下関係を保持し、関係グラフが循環しないこと |
| 保証の意味 | 数学的形式証明ではなく、固定されたモデル・入力・補間方式・許容誤差・検証器に対する数値工学的保証 |
| 保証しないもの | 紙の厚み、弾性、塑性、摩擦、破れ、手指の到達性、人間向け折り順、現実の紙での折りやすさ |
| 設計方式 | ツリー法と、本来の格子型箱ひだを別々の生成工程として扱う |
| 対応規模 | 葉数 `2..20`。内部枝と末端枝の双方に幅を設定できる |
| 用紙形状 | 正方形および任意比率の長方形 |
| 箱ひだ近似 | 元の枝長・幅と格子値の相対誤差は各値 1% 以下。元値、格子値、誤差を記録する |
| 計算時間 | 強制タイムアウトを設けない。進捗表示とユーザーキャンセルを必須とする |
| 配布前提 | Windows 10/11 の Chrome、Edge、Firefox で動く静的 Web/PWA。外部計算サービスを使わない |

「検証済み」は、紙の物理的実現性や数学上の厳密な存在証明を意味しない。UI とエクスポートには、使用した検証器の版、許容誤差プロファイルおよびこの保証範囲を表示できる情報を残す。

## 3. 対応入力と完了結果の原則

### 3.1 3 段階の入力判定

入力を次の順で判定する。

1. **構文妥当性**: スキーマ、有限数、参照 ID、範囲、データ量が正しい。
2. **ツリー妥当性**: 連結、閉路なし、葉が 2 以上、各枝長が正、各幅が非負である。
3. **対応範囲判定**: M0F で確定する workflow 別の純粋関数 `P_generation_v(input)` または `P_fold_v(input)` が `true` を返す。

該当 predicate が `true` の入力を「対応入力」と呼ぶ。設計入力は葉数が 20 以下であることだけでは対応入力と断定せず、次数、数値精度、長さ比、幅、用紙比、格子条件、回転順序などを含める。FOLD 入力は、頂点・辺・面数、assignment、縮退、最小特徴量、面隣接等を別 profile で判定する。

以下、workflow を特定しない記述では、`P_generation_v` と `P_fold_v` を総称して `P_v` と書く。

一方、いずれの predicate も次のように定義してはならない。

- 「ソルバーが解けた入力」や「経路が見つかった入力」とする循環定義
- fixture の ID、乱数 seed、既知サンプルへの一致で通す定義
- 浮動小数点の偶然の丸め結果だけに依存する定義
- 葉数 20 の固定例 1 件だけを通し、同規模の一般的な入力をすべて除外する定義

### 3.2 対応入力に許される最終結果

キャンセルされずに完了した対応入力は、必ず次のどちらかで終了する。

- `verified`: CP、目標状態、連続折り経路、積層順序および検証証跡を独立検証器が受理した。
- `no-solution-certified`: 設定された紙、方式、格子上限等では解がないことを、検証可能な必要条件違反または有限探索証明で示した。

次の状態は M0F の調査結果としては記録できるが、v1 の対応入力に対する製品上の正常完了にはできない。

- `inconclusive`: 探索したが解も解なし証跡も得られない。
- `numerical-failure`: 特異点、残差、桁落ち等により検証できない。
- `internal-error`: 実装不具合または検証器不一致。
- `cancelled`: ユーザーが中断した。既存の検証済み結果は上書きしない。

対応入力が `inconclusive`、`numerical-failure` または `internal-error` になる場合、M0F は未達である。アルゴリズムを修正するか、理由のある事前制約として `P_v` を改訂し、全試験を再実行する。実行時間が長いことや探索回数の上限到達を `no-solution-certified` とみなしてはならない。

### 3.3 対応範囲の最低限の広さ

最終的な `P_v` は、両方式について、葉数 `2..20` の各葉数に少なくとも 1 つの連続的な入力族を含まなければならない。「連続的な入力族」とは、長さ・幅・用紙比の少なくとも一つを小さく変えても対応範囲内に留まる集合を指す。

さらに、後述する 4 葉、8 葉、20 葉、内部枝幅、長方形用紙の必須 fixture を除外してはならない。この最低条件を満たす `P_v` を確定できない場合は `NO-GO` とする。

FOLD 検証 profile は `REF-FOLD-NOFACES` と `REF-FOLD-FLAT-VERIFIABLE` を含む連続的な数値摂動族を少なくとも一つ持ち、ツリーの有無に依存せず事前判定できなければならない。

## 4. 検証対象の形式モデル

### 4.1 入力ツリー

入力は埋め込み順序付き距離木 `T = (V, E)` とする。

- `|leaves(T)|` は `2..20`。
- 各枝 `e` は長さ `l_e > 0` と幅 `w_e >= 0` を持つ。
- 内部枝にも幅を設定できる。
- 各ノードの接続枝には巡回順序を持たせる。画面上の配置から決定した順序をプロジェクトへ明示保存し、生成中に暗黙変更しない。
- 対称要素は実ノード・実枝として展開でき、対応関係を保持する。
- 入力数値の最大桁数、最小正値、最大長さ比、最大次数は M0F で決め、対応範囲プロファイルへ固定する。

生成結果には、各入力枝と、目標状態の軸領域・幅領域・関連する CP 要素との追跡可能な対応表を含める。この対応表がない CP は、見た目が似ていても目標ツリーを実現したとは判定しない。

### 4.2 CP と面複体

CP は、紙領域内の直線線分からなる平面直線グラフとして扱う。交点で線分を分割し、近接頂点を規定の方法で統合し、半辺構造等を用いて面を列挙する。

次を必須不変条件とする。

- 境界が元の長方形と一致する。
- 折り線は交点で必ず頂点化され、T 字交差や重複線分の扱いが一意である。
- 面は自己交差せず、面積が許容誤差を超え、向きが統一される。
- 各非境界辺にはちょうど 2 面、各境界辺にはちょうど 1 面が接する。
- 三角形分割は面構築後に行い、三角形分割による補助辺を折り線と混同しない。
- `vertices_coords`、`edges_vertices`、`edges_assignment`、`faces_vertices` の参照が整合する。
- CP、面、入力枝、生成上の役割、M/V、目標状態、経路証跡の ID 対応が失われない。

### 4.3 剛体折り経路

面集合を `F` とし、時刻 `t in [0, 1]` における各面の剛体変換を `X_f(t) in SE(3)` とする。検証対象の経路は少なくとも次を満たす。

- `X_f(t)` は全区間で連続する。
- 同一面内の任意の 2 頂点間距離は保存される。
- 折り線を共有する面の両側で、共有線分の 3D 位置が一致する。
- 閉路を横断して面変換を合成したときの閉合残差が許容値内である。
- `t = 0` は展開状態、`t = 1` は規定誤差内の目標平坦状態である。
- M/V 符号は、経路中の符号と最終折り角に整合する。途中で符号を変える設計を許す場合は、変更イベントを証跡へ明示する。

経路の離散フレーム列だけでは連続経路の証拠にならない。各隣接フレーム間について、採用した補間・継続モデルが全時刻で拘束を満たすことを、区間包絡、誤差上界または同等の検証可能な方法で示す。

### 4.4 接触、貫通および積層順序

非隣接面の組について、次の分類を行う。

| 分類 | 扱い |
|---|---|
| 離隔 | 許可 |
| 点・辺・面の接線接触 | 許可 |
| 同一平面上の面領域の重なり | 積層順序が定義できれば許可 |
| 横断的な交差 | 禁止 |
| 接触中に面の上下が反転する通り抜け | 禁止 |
| 共有頂点・共有辺での隣接面接触 | ヒンジ位相に従い許可 |

最終状態の重なり面には上下関係を付ける。上下関係は重なり領域ごとに定義し、必要なら面を join 辺で細分して一つの面対が複数の不整合な重なり領域を持たないようにする。積層グラフに循環がある場合は失敗とする。

ゼロ厚モデルでは「距離が小さい」だけで接触と貫通を区別できない。向き付き幾何述語、接触直前・直後の側判定および積層順序の履歴を組み合わせる。許容誤差を大きくして交差を接触扱いする実装は禁止する。

### 4.5 目標ツリーとの一致

目標状態の合否は、単に全体が平らであることでは判定しない。入力枝との対応表を用いて、少なくとも次を測定する。

- 木の接続関係と分岐の巡回順序
- 枝ごとの軸長と入力長さ比
- 末端枝・内部枝それぞれの有効幅
- 枝間の不要な融合または欠落の有無
- 紙面外への逸脱の有無
- 面の最終 `z` 座標範囲と最終折り角

長さと幅の判定量の定義は、生成方式ごとに図付きで固定する。測定点を結果に都合よく選べないよう、入力枝から測定対象を決める規則を先に定義する。

### 4.6 ツリーを持たない FOLD の目標

FOLD プロジェクトは入力ツリーを推定しない。検証目標は、現在の CP 幾何を変えず、`M`/`V` の符号を尊重し、`F` を剛体的に接続し、`U` を証跡内で `M`/`V`/`F` のいずれかへ明示解決した、無矛盾な完全平坦状態とする。

- 目標 face transform、解決後 assignment、overlap region、積層順序を `ImportedFoldTargetState` として保存する。
- import 元の `faceOrders` と custom metadata は候補に利用してよいが、正しさを信用しない。
- FOLD 検証用 `SupportProfile` を生成用 profile から分離する。
- `P_fold_v` を通った入力だけ、キャンセルされない限り `verified` または `no-solution-certified` へ停止する契約を持つ。範囲外・探索中でも表示、M/V 編集、警告付き出力、未検証候補 3D は許可する。

## 5. 「本来の箱ひだ」の操作的定義

M0F では、葉配置だけを格子点に丸めてツリー法の共通 molecule へ渡す方式を箱ひだとは認めない。箱ひだ生成は独立の構成工程を持ち、次をすべて満たす。

1. 紙上に正方形セルからなる直交格子を定義し、長方形では横 `Nx`、縦 `Ny` を別管理する。
2. 主要頂点と折り線方向を格子に量子化する。使用可能な方向族は水平、垂直および 45 度対角を基本とし、例外を認める場合は箱ひだ上の役割と必要性を仕様化する。
3. axial、ridge、hinge に加え、必要な axial+N（整数 elevation の等高線）または同等の格子型 pleat routing を生成する。
4. 分岐、river、幅付き枝および junction を、単なる空白領域ではなく折り畳める crease gadget として閉じる。
5. 元の枝長・幅、格子値および各相対誤差を出力し、各誤差を 1% 以下にする。元値 `v > 0` の相対誤差は `abs(gridValue - v) / v` とし、元の幅が `0` の場合は格子値も厳密に `0` とする。
6. 生成した全 CP に対して、面構築、M/V、目標状態、積層順序および連続折り経路の検証を行う。

任意比率の長方形で正方形セルを保つ方法は M0F で確定する。候補は、`paper.width / Nx` と `paper.height / Ny` の差を許容範囲内にする格子比近似、または正方形セル領域と明示的な境界処理の併用である。セルを長方形へ引き伸ばして「正方形格子」と呼ぶことは禁止する。採用方式、用紙比誤差、最大 `Nx`・`Ny` は対応範囲プロファイルへ記録する。

## 6. 検証課題と候補アプローチ

以下は試す価値のある候補であり、採用アルゴリズムを先に断定するものではない。各候補は同じ fixture と独立検証器で比較する。

### 6.1 ツリー法 CP 生成

候補:

- circle/river packing とパス条件から有効領域を構成し、適合する polygon/molecule を充填する。
- universal molecule を、適用可能性を明示判定した領域だけに使い、適用外は別 molecule または対応範囲外とする。
- 最終面の向きと非循環な facet ordering graph から、M/V を構成する。
- 経路が既知の局所 gadget の組合せに生成範囲を限定し、構成と同時に経路証跡を作る。

universal molecule から CP と平坦な folded form が得られることだけでは、展開状態から面を曲げずに到達できる連続経路を保証しない。既知の剛な反例を adversarial fixture に含め、適用範囲を過大評価しない。

### 6.2 箱ひだ CP 生成

候補:

- 整数格子上の polygon packing と elevation contour を構築し、axial+N を選択して pleat network を作る。
- 分岐次数、枝幅、向きの有限パターンごとに、経路検証済み junction gadget を合成する。
- 格子上の配置、pleat routing、積層順序を制約充足問題として同時に解く。
- `Nx`・`Ny` の有限候補を誤差順に列挙し、1% 条件と構成可能性を満たす最初の証跡付き解を採用する。

離散配置が可能でも、junction の折り経路または積層が成立するとは限らないため、配置成功を CP 生成成功として扱わない。

### 6.3 平面配置と FOLD 面構築

候補:

- 適応精度の向き判定・交差判定を使った線分 arrangement
- DCEL/半辺構造による有界面列挙
- 入力を整数格子または有理数へ正規化できる箱ひだ部分での正確計算
- 浮動小数点フィルタと高精度 fallback の組合せ

固定 epsilon だけで交差・同一点・線上判定を行う方式は、位相が入力スケールや処理順に依存するため採用しない。

### 6.4 目標状態と M/V・積層の構成

候補:

- CP の双対グラフをたどる剛体変換伝播と閉合拘束
- facet orientation と reduced/facet ordering graph を用いた M/V 構成
- 目標 uniaxial tree から面の目標変換と重なり順序を逆算する構成
- 複数の M/V 分岐を有限列挙し、経路検証に通るものだけを採用する方式

前川・川崎条件は局所検査として残すが、それだけを大域的な平坦折り、剛体折り経路または非貫通の証拠にしてはならない。

### 6.5 連続折り経路の生成

候補:

- 生成 gadget に付随する構成的な折りスケジュールを合成する。
- 目標側から展開側へ、拘束ヤコビアンを監視しながら数値 continuation を行う。
- 独立自由度だけを探索変数にし、従属折り角を拘束式から復元する RRT/PRM 系の configuration-space planning を行う。
- 構成的経路を初期解とし、局所 continuation と衝突回避で補正するハイブリッド方式を使う。

完全平坦な初期・終端は特異点や分岐点になり得る。初期微小運動の選択、終端への極限処理、分岐の識別方法を個別に検証する。単純な全折り線同率補間は候補経路の一つにも固定しない。

### 6.6 連続自己衝突判定

候補構成:

- BVH 等による swept volume の broad phase
- 非隣接要素に対する vertex-face と edge-edge の narrow phase
- 適応精度述語または区間演算による符号確定
- 接触時刻の前後での側判定と積層順序更新
- 非線形な剛体回転軌道を覆う conservative advancement または検証付き時間細分

線形頂点補間向け CCD を使う場合、その補間が剛体面の実際の区間軌道と一致するか、実軌道を安全側に包絡することを示す。剛体回転する面を端点間で単に線形補間すると面剛性が崩れるため、そのまま保証器へ流用してはならない。

### 6.7 独立検証器

生成器とは別の入口とデータフローを持つ reference verifier を作る。reference verifier は CP と経路証跡だけを入力とし、生成器の内部キャッシュや「成功」フラグを信用しない。

最低限、次を独立に再計算する。

- CP 位相と面列挙
- 各区間の面剛性、ヒンジ一致、閉合および連続性
- 初期・目標状態と入力ツリー対応
- 全区間の非貫通と接触分類
- 最終積層順序の整合性と非循環性
- tolerance profile と入力・CP・経路のハッシュ一致

生成器と検証器が高レベルの制約実装を共有し、同じバグで合格する構成は禁止する。低レベル幾何述語を共有する場合は、別実装との differential test または既知の正確解で検証する。reference verifier は生成器より遅くてよい。

## 8. 数値許容誤差の決定と検証

### 8.1 許容誤差プロファイル

短辺を `1` に正規化し、少なくとも次を別々に定義する。全項目を一つの `epsilon` で代用しない。

| キー | 対象 |
|---|---|
| `coordMergeAbs` | 2D 頂点統合 |
| `orientPredicate` | 向き・交差判定の高速フィルタと高精度 fallback 条件 |
| `minFaceArea` | 退化面判定 |
| `rigidDistanceAbs` / `rigidDistanceRel` | 面内距離保存 |
| `hingeCoincidenceAbs` | 共有折り線の 3D 一致 |
| `loopClosureAbs` / `loopClosureAngle` | 閉路拘束 |
| `pathContinuityAbs` / `pathContinuityAngle` | 区間境界の連続性 |
| `goalFlatnessAbs` / `goalFoldAngle` | 目標平坦状態 |
| `treeLengthRel` | 入力枝長との一致 |
| `treeWidthRel` | 入力枝幅との一致 |
| `gridQuantizationRel` | 箱ひだの値の量子化。上限は `0.01` |
| `contactClassification` | 接触・横断・側反転を分類する判定方式 |
| `timeCoverage` | 区間検証の被覆と端点重なり |
| `displayEndpointOffset` | 3Dで完全平坦直前を表示する場合の経路時刻offset。検証判定には使わない |

`contactClassification` は単なる距離 epsilon にしない。可能な限り正確な符号述語を使い、符号を確定できない場合は成功ではなく高精度 fallback または `numerical-failure` とする。

### 8.2 具体値の決め方

1. IEEE 754 double を基準に、正確解を持つ小規模例で丸め誤差を測る。
2. 入力全体の平行移動、反転、回転、頂点順変更、同一スケールへの正規化で結果が変わらないことを確認する。
3. 各対応範囲境界で、入力値を 1 ULP から実用刻みまで摂動する。
4. 通常精度と高精度 reference の差を測る。
5. tolerance を半分・2 倍にした感度分析を行い、合否が不安定な領域を `P_v` から安全余裕付きで除外する。
6. 最も悪い観測誤差に安全係数を掛けた候補を定め、別 seed の holdout fixture で確認する。
7. 値、根拠、基準環境、検証器版を `tolerance-profile-v1.json` に凍結する。

合格しやすくするために fixture ごとに tolerance を変えてはならない。変更する場合はプロファイルを版上げし、全 fixture と mutation test を再実行する。

### 8.3 「100% 平坦」の扱い

理論上の目標は平坦状態とするが、数値判定は `goalFlatnessAbs` と `goalFoldAngle` を用いる。3D 表示は数値安定性のため経路の終端直前を描画してもよいが、検証器は別途、終端の目標拘束と積層順序を検証する。

### 9.4 接触と退化の境界例

次を個別に用意する。

- 点接触、辺接触、面の接線接触
- 最終区間で初めて生じる合法な同一平面重なり
- 同一時刻に複数接触が起きる例
- ほぼ平行な面、ほぼ共線な辺、ほぼ一致するイベント時刻
- 合法な接触の直後に離れる例と、不合法に通り抜ける例
- 対称性により経路が分岐する例
- 完全平坦終端へ近づくほど条件数が悪化する例

### 9.5 生成的・境界試験

固定 fixture だけで対応範囲を決めない。最終候補 `P_v` に対し、最低限次を実行する。

- 両方式、各葉数 `2..20` について、固定 seed で 20 件以上の対応入力を生成する（計 760 件以上）。
- FOLD検証profileについて、REF-FOLD-FLAT-VERIFIABLEの幾何/MVをprofile内で摂動した対応入力を固定seedで20件以上、各数値境界の内外を各5件以上生成する。
- 各 `(方式, 葉数)` で、対応範囲の各数値境界に近い例を少なくとも 5 件含める。
- 次数分布、長さ比、幅、用紙比、対称・非対称を層別化する。
- 小規模で有限列挙できるクラスは全列挙する。
- 生成に使っていない seed の holdout 集合を最終判定に使う。

生成的試験はバグ検出と範囲推定の証拠であり、無限個の実数入力に対する証明の代わりにはしない。対応範囲全体への主張には、構成法の成立条件、有限探索の完備性または明示した安全余裕の説明を併記する。

## 10. 対応入力 predicate の確定方法

### 10.1 候補特徴量

少なくとも次を候補制約として評価する。

- 共通: 葉数、最大次数、次数列、巡回順序の有無、枝数
- 数値: 入力小数桁、最小正長、最大/最小長比、幅/長さ比、幅の分岐収容条件
- 紙: 縦横比、正規化後の最小特徴量、境界余裕
- ツリー法: packing/molecule の適合条件、必要な分子型、特異配置からの距離
- 箱ひだ: `Nx`・`Ny`、セル正方形性、方向族、量子化誤差、junction gadget の対応表
- 経路: gadget 合成条件、自由度上限、閉路拘束の条件数、必要な積層パターン

### 10.2 確定手順

1. 文献と構成法から、成功のための十分条件候補を列挙する。
2. その条件を solver 実行前に有限時間で評価できる形へ変換する。
3. 条件内・条件境界・条件外の fixture を作り、generator と reference verifier で差を確認する。
4. 最小反例を縮約し、必要ならアルゴリズムまたは条件を修正する。
5. 条件内の全入力に対する終了根拠を記述する。離散探索なら探索空間の有限上界、構成法なら各工程の適用可能性と失敗条件を示す。
6. 理由コード、ユーザー向け説明、修正提案を付ける。
7. `support-profile-v1.json`、JSON Schema、pure function `checkSupport`、テストを同じ commit で凍結する。

試作インターフェースの最低形は次とする。

```ts
type SupportDecision =
  | { supported: true; profileId: string; normalizedInputHash: string }
  | {
      supported: false;
      profileId: string;
      reasons: Array<{
        code: string;
        targetIds: string[];
        details: Record<string, number | string>;
        remediation?: string;
      }>;
    };
```

### 10.3 Predicate 変更規則

- 制約追加・緩和はいずれもプロファイルを版上げする。
- 変更時は全 fixture、holdout、mutation、ベンチマークを再実行する。
- 既存プロジェクトには評価時の profile ID を保存する。
- 対応範囲を狭める変更は ADR とプロダクト所有者の承認を必要とする。
- 20 葉対応、両方式、内部枝幅、長方形箱ひだの最低 fixture を除外する変更は M0F の `GO` 条件を失う。
- 手動 M/V 編集、CP 形状変更、依存する revision vector 成分、SupportProfile、ToleranceProfile、検証器版の変更後は以前の対応判定・検証済み状態を無効にし、再判定・再検証する。

## 11. 測定と再現性

### 11.1 基準環境

初期基準は次とし、M0F report で CPU 型番、GPU、OS build、ブラウザ版を固定する。

- Windows 11
- 4 コア CPU
- メモリ 16 GB
- 内蔵 GPU
- 1920 x 1080
- 最新版 Chrome、Edge、Firefox

### 11.2 記録する測定値

各 fixture で次を JSON Lines と集計表に残す。

- predicate 判定時間と理由
- CP 生成、面構築、目標構築、経路探索、CCD、reference verify の各時間
- p50、p95、最大時間。3 秒（箱ひだ配置）と 30 秒（ツリー法配置）に対する達否も努力目標として併記
- 探索ノード数、continuation step 数、時間細分数、fallback 高精度述語回数
- 面数、三角形数、候補衝突対数、実接触イベント数
- peak working set、Worker/WASM メモリ、成果物サイズ
- 最大の剛性、ヒンジ、閉合、目標、長さ、幅残差
- generator と reference verifier の合否一致
- seed、build hash、profile ID、ブラウザ、実行開始時刻

乱数を使うアルゴリズムは seed を保存し、失敗を完全再現できるようにする。同一入力・seed・build を 5 回実行し、最終合否と証跡ハッシュが安定することを確認する。並列実行で証跡順が変わる場合は正規化規則を定義する。

### 11.3 UI 応答とキャンセル

- 重い処理は Worker または WASM Worker で行う。
- 進捗メッセージは phase、完了量、既知の全体量、経過時間を持つ。全体量が不明なら不明と表示し、偽の百分率を出さない。
- キャンセル要求の確認応答 p95 は基準機で 500 ms 以内を初期目標とし、到達不能なら M0F report で測定根拠と新しい上限を決める。
- キャンセル後に古い job が結果を commit しないことを job ID と revision で試験する。
- 計算中も UI thread の long task を計測し、50 ms 超が継続しないことを確認する。

性能時間は保証の代わりにしない。遅くても正しく完了する入力と、速く誤判定する入力では前者だけが合格候補である。ただし必須 fixture でブラウザ OOM、クラッシュ、回復不能な UI 停止が起きる場合は `NO-GO` とする。

## 12. 成功・失敗・停止基準

### 12.1 個別実験の判定

`PASS`:

- 期待結果と generator 結果が一致する。
- reference verifier が独立に同じ結論を返す。
- 全測定値と証跡が保存され、再現できる。

`FAIL`:

- 対応入力で false `verified`、未検出貫通、拘束違反、誤った解なし、または非再現結果が 1 件でもある。
- 正例を誤拒否、または負例を誤受理する。
- 証跡が欠落し第三者が再検証できない。

`inconclusive`:

- 解の有無を決められない、数値符号を確定できない、必要データがない。

`inconclusive` は `PASS` に数えない。

### 12.2 候補アルゴリズムの停止・切替基準

次のいずれかで、その候補の探索を止め、別候補または範囲改訂へ切り替える。

- 最小化・再現された false `verified` が生じ、独立検証可能な修正方針がない。
- 経路のサンプル点だけしか検査できず、区間全体の包絡または CCD へ拡張できない。
- 合法な接触と貫通を原理的に区別できない。
- 完全平坦端の特異性により、必須 fixture の経路を連続に接続できない。
- 必須 20 葉 fixture でブラウザ資源上限を超え、静的 Web の範囲で改善案がない。
- 候補が GPL 実装の移植、非公開サービス、常時外部通信または有償専用ソルバーを必要とする。

単に所要時間が長い、乱数探索が一度失敗した、未実装箇所がある、という理由だけでは候補の不可能性を断定しない。停止時は最小反例、測定値、試した修正、残る仮説を ADR に残す。

### 12.3 M0F `GO` 条件

以下をすべて満たした場合だけ `GO` とする。

1. 循環定義でない `P_v` が確定し、3.3 の最低対応範囲を満たす。
2. 対応入力の完了結果が `verified` または `no-solution-certified` に限定される根拠がある。
3. 本来の箱ひだとツリー法の別生成工程が、各必須 fixture を通る。
4. 幅付き内部枝、長方形、固定および holdout の 20 葉 fixture が通る。
5. CP から整合した面を構築し、FOLD 再読込で位相・ID・積層が保たれる。
6. 初期から目標までの全時間区間に、剛体拘束を満たす連続経路の証跡がある。
7. 合法な接触・重なりを許し、面貫通、順序反転、積層循環を検出する。
8. reference verifier と mutation test が合格する。
9. tolerance profile が感度試験を経て凍結され、保証文言とともに表示可能である。
10. Windows の 3 ブラウザで静的・ローカル計算として動き、進捗・キャンセル・再現性が確認される。
11. fixture、生成試験、holdout、境界試験が全件合格し、未解決の重大不具合がない。
12. ライセンスと出典監査が完了している。
13. `runtime-limits-v1.json` が凍結され、ファイル bytes、頂点・辺・面・文字列、`Nx/Ny`、履歴 byte budget、Worker キャンセル応答、入力桁数・最小特徴量・最大比率に未定値がない。
14. 製品へ引き継ぐdomain artifact schemaに、未解決の`unknown`、汎用`constraints`/evidence payload、未選択の経路表現、未定上限が残っていない。runtime validation前のraw外部入力を`unknown`で受けることと、保持対象custom fieldを`JsonValue`で表すことは例外とする。

制約付きの `P_v` であっても、3.3 を満たし制約を入力前に明示できるなら `GO` としてよい。未決定項目を「実装中に決める」として残した条件付き承認は行わない。

### 12.4 M0F `NO-GO` 条件

次のいずれかが残る場合は `NO-GO` とする。

- `P_v` が必須範囲を満たさない、またはソルバー成功に依存する。
- 20 葉対応を固定例だけに限定し、入力族として保証できない。
- 箱ひだが格子配置したツリー法に留まり、5 章を満たさない。
- universal molecule、前川・川崎、視覚シミュレーションのいずれかだけを折り経路保証とする。
- 経路の離散サンプル間に未検査区間がある。
- 接触を全禁止することで最終重なりを避ける、または接触許容 epsilon で貫通を見逃す。
- reference verifier を用意できない。
- 必須 fixture に false `verified`、誤った解なしまたは再現不能が残る。
- 静的 Web/PWA、無料配布、許可ライセンスの条件内で成立しない。

`NO-GO` は「折り紙アプリ全体が不可能」という意味ではない。保証範囲、葉数、方式、物理モデルまたは配布方式のいずれかを再協議し、要件を改訂しなければ製品実装へ進めないという判定である。

## 13. M0F の必須成果物

`GO` 判定時に次をリポジトリへ保存する。配置先は M0F 開始時に決める。

6. `no-solution-certified` の判別可能な証跡 schema と reference checker
7. 独立 reference verifier と mutation test
8. fixture manifest、正例、負例、holdout、provenance
9. 生ベンチマーク JSONL、集計表、基準環境情報

## 14. 製品実装へ引き継ぐ契約

M0F の `GO` 後、製品実装は少なくとも次の境界を維持する。次の型名と discriminant は `02_技術設計書.md` と一致させる。`unsupported`、`invalid`、`internal-error` は対応範囲判定または異常系であり、下記の対応入力用 terminal には含めない。

```ts
type SupportedSolveTerminal =
  | {
      kind: 'verified';
      solution: {
        layout?: LeafLayout;
        creaseMesh: CreaseMesh;
        fold: FoldFile;
        target: TargetState;
        pathCertificate: PathCertificate;
        verification: VerificationRecord;
      };
    }
  | {
      kind: 'no-solution-certified';
      certificate: NoSolutionCertificate;
    }
  | { kind: 'cancelled' };
```

- `verified` は reference verifier の成功後にだけ生成する。
- ファイルに保存された `verified` 文字列を信用せず、canonical data からハッシュを再計算し、検証器版、プロファイル版、証跡内容を reference checker で照合する。
- 入力、用紙、格子、対称、配置、CP、M/V の変更で検証状態を `stale` または `unverified` に戻す。方式変更はツリーだけを維持して派生結果を破棄し、`none` とする。
- 3D スライダーは path certificate 上の時刻を表し、全折り線を同率で動かさない。
- ユーザーがキャンセルした場合、最後の検証済み結果を維持する。
- 未検証 CP の表示・編集・警告付き出力は許すが、「折り可能性検証済み」と表示しない。

## 15. 主要リスクと M0F での扱い

| リスク | M0F で必要な対応 |
|---|---|
| 一般の剛体折り可能性は計算困難 | 対応範囲を構成的に限定し、成功後判定ではない predicate を固定する |
| universal molecule に連続剛体経路のない例がある | 既知反例を fixture 化し、適用可能性または別構成を検証する |
| 局所平坦折り条件と大域経路を混同する | 前川・川崎とは別に経路・衝突・積層を検証する |
| 平坦始点・終点の特異性 | 分岐選択、終端処理、区間証跡を試験する |
| ゼロ厚接触の数値退化 | 適応精度述語、側判定、積層履歴、adversarial fixture を使う |
| 経路 planner の非完備性 | 構成法または有限探索で終了根拠を持つ範囲だけ `P_v` に含める |
| 箱ひだを配置問題へ矮小化する | axial+N、pleat routing、junction、格子方向を合格条件にする |
| generator と verifier の共通バグ | 独立データフロー、differential test、mutation test を必須にする |
| ブラウザのメモリ・長時間計算 | 20 葉で計測し、Worker/WASM、キャンセル、資源解放をゲートで確認する |
