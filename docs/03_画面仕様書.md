# OriDesign（仮称）画面仕様書 v1.1

更新日: 2026-07-14 / ステータス: 仕様確定（本番実装はM0F Go後）

Windows 10/11のデスクトップブラウザ向けシングルページPWA。通常利用画面は3ペイン構成とし、画面幅1280px以上を対象とする。1280px未満では機能を隠さず、画面上部に「画面幅1280px以上を推奨します」を表示する。モバイル専用レイアウトは作らない。

本書で「検証済み」とは、厚さゼロの数値モデルにおいて、平面状態から、設計projectでは目的の平坦なuniaxial base、FOLD projectでは現在のCP/M/Vに整合する完全平坦目標まで、面の貫通がなく、接触・重なり・積層順序を含む連続折り経路が表示中の許容誤差で確認済みであることを指す。数学的な形式証明や実物の紙での可折性保証を意味しない。

## 1. 全体レイアウト

```text
┌──────────────────────────────────────────────────────────────┐
│ ヘッダー: ロゴ | 種別/名前/保存状態 | ファイル | Undo/Redo   │
│           検証状態 | 生成・検証 / 再検証 | 設定             │
├──────────────┬────────────────────────┬─────────────────────┤
│ 左: 設計      │ 中央: メインビュー      │ 右: プロパティ      │
│ 320px         │ 可変                    │ 320px               │
└──────────────┴────────────────────────┴─────────────────────┘
│ ジョブドロワー: 工程 / 進捗 / 経過時間 / キャンセル          │
└──────────────────────────────────────────────────────────────┘
```

- 設計プロジェクトの中央タブは「ツリー」「配置」「展開図」「3D」の4つ。
- FOLDプロジェクトの中央タブは「展開図」「3D」の2つ。ツリーを推定・捏造しない。
- ヘッダーのプロジェクト名の左に「設計」または「FOLD」を表示する。
- プロジェクト名の右に、ファイル保存前または保存後に変更があれば「未保存」を文字と●アイコンで表示する。
- 「ファイル」メニューは「新規」「開く」「上書き保存」「名前を付けて保存」「FOLDをインポート」「最近使ったプロジェクト」を持つ。
- 主操作ボタンは、設計プロジェクトでは「生成・検証」、FOLDプロジェクトおよび手動M/V編集後では「折り経路を検証」とする。実行できない場合もボタンを消さず、disabled理由を隣接テキストとツールチップに表示する。

## 2. 状態の表示規則

### 2.1 検証状態

ヘッダー、左ペイン、展開図、3Dの全てで同じ語とアイコンを使う。

| 状態 | 表示 | 意味と操作 |
|---|---|---|
| `none` | 未生成 | 現revision vectorに対応するCPがない。「生成・検証」を案内する |
| `stale` | 要再生成 | 入力変更前のCPが残っている。閲覧・警告付き出力は可能だが検証済みとは表示しない |
| `unverified` | 未検証 | CPはあるが、有効な折り経路がない。検証失敗理由があれば併記する |
| `verifying` | 検証中 | Workerジョブ実行中。工程、進捗、経過時間、キャンセルを表示する |
| `verified` | 検証済み | CPと折り経路が依存するrevision vector成分とhashが一致し、数値検証に合格している |
| `no-solution-certified` | 解なし | 対応範囲内だが解が存在しないことを確認済み。独立検査可能な根拠を表示する |
| `unsupported` | 対応範囲外 | M0Fで確定する対応範囲判定に不合格。該当条件と変更候補を表示する |
| `error` | 計算失敗 | 入出力不良または内部エラー。解なしとは扱わず、再試行用情報を表示する |

- `input-invalid`（未接続、閉路、葉不足、0以下の長さなど）は検証状態とは別の入力バッジ「入力不正」で表す。編集中は許容するが「生成・検証」は無効にする。
- 色だけに依存せず、状態名、形の異なるアイコン、必要に応じて線種を併用する。
- `verified`の詳細には対象revision vector、入力/CP/経路hash、検証日時、検証エンジン版、SupportProfile、ToleranceProfile、最小離隔、接触数、積層順序の整合結果を表示する。紙の短辺を1とする正規化座標系で値を示す。具体的な許容誤差はM0F成果物を参照する。
- キャンセルは永続状態にしない。「キャンセルしました。以前の結果を維持しました」と通知し、直前のCP・検証済み経路・状態を一切上書きしない。

### 2.2 検証詳細パネル

状態バッジを押すと右ペインに次を工程順で表示する。

1. 入力: ツリー構造、枝順、用紙、方式、対応範囲。
2. 配置: パス条件、用紙内包含、箱ひだ格子近似。
3. CP: 平面グラフ、面、前川条件、川崎条件。
4. 経路: 開始・終端条件、連続自己衝突判定、面の貫通、接触、積層順序。
5. 判定: 検証済み、解なし、対応範囲外、または失敗の根拠。

各問題行は重大度、対象ID、数値、許容値、「該当箇所へ移動」を持つ。川崎違反はM/V変更では修正できない旨を表示し、自動生成結果では生成失敗または未検証の下書きとして扱う。前川違反にはM/V編集への導線を出す。

## 3. 左ペイン: 設計

FOLDプロジェクトではこのペインを「FOLD情報」に切り替え、ファイル名、頂点・辺・面数、面再構築の有無、検証状態、非対応警告だけを表示する。

設計プロジェクトでは次を上から配置する。

### 3.1 方式と用紙

- 方式: 「箱ひだ」「ツリー法」のセグメントコントロール。
- 方式変更の確認ダイアログには「ツリーは維持し、配置・CP・M/V編集・検証結果を破棄します」と明記する。「切り替える」「キャンセル」を用意する。
- 箱ひだへ切り替えた直後は格子候補を自動計算し、採用までは生成を無効にする。
- 用紙: 「正方形」プリセットと「長方形」。幅比・高さ比を正の実数で入力する。mm等の実寸入力は設けない。
- 用紙プレビューには「比率指定。PDFでは選択した用紙へ自動フィット」と表示する。

### 3.2 箱ひだ格子

箱ひだ方式だけに表示する。本来の格子型箱ひだ専用の生成工程であり、ツリー法の葉配置を格子に制限したものではない。

- 列数`Nx`、行数`Ny`を別々の整数入力として表示する。`Nx=4`または`Ny=4`も許可する。上限はM0F成果物の値を使用する。
- `paper.width / Nx`と`paper.height / Ny`がM0Fの許容誤差内で等しい正方形セルになる組だけを有効とする。
- 「自動提案」は上位3件を、`Nx × Ny`、セル寸法、最大近似誤差、推定計算量とともに表示する。保証条件を満たす候補だけを「採用可能」にする。
- 枝の長さと幅は実数のまま保存し、格子値へ暗黙に書き換えない。「近似」テーブルに枝ラベル、属性（長さ/幅）、入力値、格子値、相対誤差を表示する。
- 全属性の相対誤差が1%以下なら合格。1%を超えた場合は、より細かい格子候補を示す。対応格子上限内に合格候補がなければ「解なし」、必要格子が対応上限外なら「対応範囲外」と区別する。
- 格子線の表示/非表示を切り替えられるが、出力幾何には影響しない。

### 3.3 対称、枝、入力検証

- 対称モード: 縦軸対称トグル。ON時は軸を表示し、片側のノード・枝と`mirrorOf`で対応する実要素を作る。軸上要素は自己対称として扱う。
- OFF時は左右両方の実要素を残す。OFF後の最初の独立編集を説明する通知を1回表示する。
- 枝テーブル: 列は「枝順」「ラベル」「長さ」「幅」。長さは正の実数、幅は0以上の実数で、内部枝にも設定できる。行選択とキャンバス選択を双方向同期する。
- 枝順はノード周りの巡回順を表し、設計意味を持つ。キャンバス配置から更新した順序を明示表示・保存し、同角度などで順序を確定できない場合は手動並べ替えを要求する。
- フッターに「ノードを追加」「枝を追加」を用意する。
- 常時検証: 連結、閉路なし、葉2以上、正の枝長、0以上の幅、枝順、対称対応、葉20以下、およびM0Fの対応範囲条件。違反は一覧とキャンバスの両方に表示する。

## 4. 中央タブ1: ツリー

- SVGキャンバス。ツールバーの「選択」「ノード追加」「枝追加」を基本操作とし、空白ダブルクリックでもノード追加、ノード間ドラッグでも枝追加できる。
- ノードドラッグは表示位置とノード周りの枝の巡回順に影響する。ドラッグ1回をUndo履歴1件にまとめる。
- 枝上にラベル、長さ、幅（0より大きい場合）を表示する。ダブルクリックまたはEnterで編集する。
- 削除はDeleteキー、右クリックメニュー、右ペインの削除ボタンから行える。削除後に一時的に不正なツリーとなることは許容する。
- 対称モードでは編集側を通常表示、従属側を半透明表示する。従属側の直接編集時は対応する編集側へフォーカスを移す。軸上へのスナップ操作を用意する。
- ノードを選択すると、接続枝を保存済み巡回順で番号表示する。曖昧な順序は警告アイコンを付ける。
- キーボード操作: Tabでツール/要素移動、Enterで選択・編集、矢印で選択ノードを移動、Shift+矢印で大きく移動、Deleteで削除、Escで操作取消。全ポインタ操作にボタンまたはキーボードの代替を用意する。

## 5. 中央タブ2: 配置

- 用紙矩形、葉、アクティブパス、幅付きriver/カドの占有領域を表示する。
- ツリー法は葉を円形マーカー、箱ひだは格子点マーカーと箱ひだの軸・river候補で表示する。箱ひだ表示は専用構成結果であり、共通molecule表示へ置換しない。
- 「自動配置」を押すとWorkerジョブを開始する。進捗、現在の探索工程、経過時間、現在の最良値、キャンセルを表示する。3秒（箱ひだ）/30秒（ツリー法）は努力目標として経過時に通知するだけで、強制終了しない。
- 葉を手動ドラッグできる。箱ひだは有効な格子点にスナップする。パス条件違反ペアを赤い波線と警告アイコンで結び、違反量をツールチップと検証詳細に表示する。
- 等号成立したアクティブパスは緑の二重線で表示する。色を識別できない場合に備え、線種を必ず併用する。
- ステータス行: スケール`s`、充填率、配置済み葉数、違反数、箱ひだ最大近似誤差。
- 手動配置は不正状態でも保存できるが、違反がある間はその配置を用いたCP生成を無効にし、再配置の理由を表示する。

## 6. 中央タブ3: 展開図（CP）

### 6.1 表示と編集

- 山=赤実線、谷=青破線、境界=黒太線、フラット=灰細線を既定とする。設定変更後も線種と記号`M/V/B/F`を併用する。
- ズーム、パン、全体表示、格子表示（箱ひだ）、参照点の正規化座標表示を提供する。ズームはホイールに加えてボタンも用意する。
- CP上部に検証状態バナーを常設する。`stale`では「この展開図は現在の設計と一致しません」、`unverified`では「折り可能性は未検証です」を表示する。
- 前川違反頂点、川崎違反頂点、非多様体、閉じない面、衝突区間を異なるアイコンでオーバーレイする。
- M/V編集モードでは折り線を選び、右ペインまたはキー`M`/`V`で割り当てる。クリック循環だけに依存しない。変更は安定した折り線IDに対するoverrideとして保存する。
- M/Vを1本でも変更した時点で、検証済み経路と積層順序を無効化して`unverified`にする。「再検証」成功まで検証済みへ戻さない。川崎条件は角度条件のためM/V編集では直せないことを表示する。
- Tab/Shift+Tabまたは折り線一覧で折り線を選択できる。選択線の端点、割り当て、役割、局所検証結果を読み上げ可能なテキストで右ペインに出す。

### 6.2 FOLDプロジェクト

- `.fold`読込直後は常に`unverified`。CP表示、M/V編集、再出力、平面状態の3D表示を許可する。
- 正式対応はFOLD 1.1/1.2のtop-level key frameにある2D平面CPで、`file_frames`が存在しないか空配列のもの。`faces_vertices`がなければ読込時に面を再構築し、その事実を履歴とFOLD情報に表示する。
- 1件以上の`file_frames`を持つ複数フレーム、初期状態が3D、非平面、非多様体、assignment `C/J`、参照不正は読込を完了せず、非対応理由と問題箇所を表示する。
- FOLD内のcustom verification metadataは「外部メタデータ（未信頼）」として表示できるが、状態復元には使わない。読込後に現在のCPと証跡をローカル検証した場合だけ`verified`にする。
- FOLDプロジェクトはツリーなしのまま`.orid.json`へ保存でき、source名/spec版/hash、保持可能なcustom field、正規化済みFOLD 1.2、M/V override、再構築面、目標状態、候補/検証済みPathCertificate、CCD/積層の証跡blob、検証結果を保持する。

### 6.3 エクスポート

「エクスポート」モーダルでFOLD/SVG/PNG/PDFを選ぶ。

- FOLDは1.2を既定出力とし、FOLD/SVGは正規化座標を保持する。
- PNG既定値は長辺4096px、入力可能範囲は実装上限まで（最大16384px）、背景は白/透明を選択可能。
- PDFは1ページ、余白10mm、A4を既定とし、A3/レターを選択可能。縦横は自動選択し、用紙中央へ最大フィットする。原寸出力ではない。
- `verified`以外も出力可能。ただし状態、理由、対象revision vector、許容誤差を確認画面に表示し、「未検証であることを理解して出力」に明示同意を求める。
- FOLD/SVG/PNG/PDFには可能な範囲で検証状態メタデータを付ける。メタデータを確実に保持できないPNG/PDFには、未検証またはstaleであることを示す小さな可視フッターも付ける。
- 書出し失敗では保存済み/未保存状態を変えない。

## 7. 中央タブ4: 3D

- Three.jsビューポート。左ドラッグ=回転、ホイール=ズーム、右ドラッグ=パン。各操作にボタンと「ビューをリセット」を用意する。
- スライダーは一律の折り畳み率ではなく、検証済み連続折り経路の時間位置`t=0–100%`を表す。折り線ごとに速度・開始時刻が異なる。
- 「先頭」「逆再生」「再生/一時停止」「末尾」と任意時刻seekを提供する。すべてPathCertificateと同じ補間を使い、表示専用の別補間で経路を変えない。
- 数学上の100%は目的の完全平坦状態。数値安定性のため3D描画は`100%−ε`で止め、ラベルは「100%（平坦状態直前を表示）」とする。`ε`はM0Fの検証プロファイルに従う。
- 経路がない未検証CPは平面メッシュを表示する。未検証の候補経路がある場合は、利用者が「未検証候補を表示」を明示選択したときだけ、警告色・透かし・専用スライダーで表示できる。検証済み経路と同じ外観や既定再生を使わず、「折り経路を検証」を常設する。
- 表面/裏面を2色で表示し、重なり時は検証済み積層順序を使う。接触は許可し、面の貫通はエラーとして区間と面IDを表示する。
- ステータスには時刻、収束値、最小離隔、接触数、積層順序の状態を表示する。
- 三角形1000枚で30fpsを必須性能とし、2000枚は努力目標。検証計算時間とは分けて表示する。
- 3Dビューの完全なキーボード代替が難しい操作は「既知のアクセシビリティ制限」からテキストの面/時刻/検証情報へアクセスできるようにする。

## 8. 右ペイン: プロパティ

選択コンテキストに応じて切り替える。

- ノード: ラベル、座標、軸上/ミラー対応、接続枝の巡回順。
- 枝: ラベル、長さ、幅、接続先、巡回順、削除。幅は葉枝と内部枝の両方で編集可能。
- 葉配置: 座標/格子点、関係するパス条件、手動配置を戻す。
- 折り線: 安定ID、端点、役割、M/V/F/B、override有無、局所検証。
- 面: 頂点、表裏、積層順位、接触/衝突相手。
- 非選択: プロジェクト統計、revision vector、成果物ごとの依存revision/hash、葉数、枝数、面数、検証概要。

段折りの段数/深さはv1に表示しない。Phase 2で追加する。

## 9. ジョブドロワーと非同期処理

- 配置、CP生成、面構築、折り経路探索、連続衝突判定、積層順序検証を工程として表示する。
- 固定タイムアウトは設けない。進捗とキャンセルを常時提供し、努力目標超過時は「計算を継続しています」を表示する。
- 各ジョブはjob ID、開始時のrevision vector、意味的入力hashを表示用データにも持つ。完了時に必要なrevision成分とhashが一致する場合だけ結果を反映する。
- 実行中に入力が変わった場合はジョブをstaleとしてキャンセルし、戻ってきた古い結果を破棄する。
- 下書きCPが得られても検証が失敗した場合は、理由を添えて`unverified`として採用するかをユーザーに確認する。自動的に最後の検証済み結果を置き換えない。
- 「解なし」はM0Fで定義する確認可能な根拠が得られた場合だけ表示する。探索中断、反復上限、キャンセル、例外を解なしにしない。

## 10. 保存、読込、復旧

### 10.1 保存

- File System Access APIでファイルハンドルを保持できるブラウザは同じファイルへ上書き保存する。非対応ブラウザは毎回ダウンロードし、「ダウンロードを確認しました」「未保存のまま」の確認を表示する。
- `Ctrl+S`はヘッダーの「上書き保存」と同じ動作。上書きAPI成功またはダウンロード後の利用者確認が得られた場合だけ未保存表示を解除し、ダウンロード開始だけでは解除しない。
- `.orid.json`と`.fold`は実行時スキーマ検証を行う。巨大ファイル、重複ID、非有限数、範囲外座標、不正参照を拒否し、JSONパスと理由を一覧表示する。サイズ・要素数上限はM0F成果物を使う。

### 10.2 自動復旧

- 各プロジェクトにUUIDを付け、編集停止2秒後にIndexedDBへスナップショットを保存する。仮レコードを書いた後、同一transactionでcurrent pointerを切り替え、途中書込み時は直前の完全snapshotを残す。
- 最近使った一覧は`updatedAt`降順の10プロジェクトとする。11件目以降は最終更新が最古で、open/lock中でないsnapshotから削除する。保護対象しかない場合は削除を延期できるが一覧表示は10件に限り、その理由を設定に表示する。ファイル保存後も復旧用データを残し、「最近使ったプロジェクト」から個別削除できる。
- 起動時に復旧候補があれば勝手に復元せず、名前、更新日時、元ファイル、検証状態を示すダイアログで「復元」「破棄」を選ばせる。
- リロード/終了前は未保存変更を`beforeunload`で警告するが、自動スナップショットの完了を保存成功とはみなさない。

### 10.3 複数タブ

- UUID単位で編集ロックを共有する。同じプロジェクトを後から開いたタブは読み取り専用となり、上部バナーに「別のタブで編集中」と表示する。
- 「こちらで編集を引き継ぐ」を押すと確認後にロックを移し、元のタブを読み取り専用へ切り替える。元のタブにも通知する。
- heartbeat消失時も自動で編集開始せず、引継ぎ確認を表示する。両タブに異なる未保存snapshotがある場合は、名前、更新日時、検証状態を並べて「このタブを採用」「別タブを採用」「キャンセル」を選ばせ、自動mergeしない。
- 読み取り専用タブでは表示、3D操作、別名保存は許可し、設計編集、M/V編集、上書き保存、ジョブ開始を無効にする。

## 11. PWA更新

- 新しいService Workerが待機しても作業中に自動更新しない。「新しいバージョンがあります。保存後に更新してください」を表示する。
- 未保存変更がなければ「更新」、未保存変更があれば「保存して更新」「後で」を表示する。保存が失敗した場合は更新しない。
- 更新後のmigrationに失敗した場合は旧snapshotを削除せず、旧版での再試行またはファイル書出しを案内する。
- PWAはビルドに含まれるアセットだけをキャッシュし、外部フォント、CDN、アクセス解析、広告、クラッシュ送信、外部APIを使わない。
- ブラウザがPWAインストールUIを提供する場合は「アプリをインストール」を表示する。提供しないブラウザでは公開URLの利用を正式経路として案内し、主要機能を制限しない。

## 12. 設定、履歴、キーボード

- 設定モーダル: CP配色、3D表裏色、3D表示設定、Undo履歴上限。
- 設定は端末共通で、プロジェクトファイルに含めない。履歴件数は既定100件、1～500件で変更可能だが、これは上限値である。M0Fのbyte budgetを先に超えた場合は未参照の古い履歴から破棄し、実保持件数と理由を設定画面に表示する。参照中の派生blobは削除しない。
- Undo対象: ツリー、枝順、用紙、方式、格子、対称、配置、M/V編集、ソルバー結果の採用。ドラッグ1回とソルバー結果の採用1回をそれぞれ1操作とする。
- Undo対象外: 配色、履歴上限、3Dカメラ等の端末設定、進捗、復旧スナップショット作成。
- 共通キー: `Ctrl+Z`=Undo、`Ctrl+Shift+Z`/`Ctrl+Y`=Redo、`Ctrl+S`=保存、Delete=選択削除、`1`～`4`=利用可能なタブ切替、Esc=モーダル/操作取消。入力欄編集中はブラウザ標準の文字編集を優先する。

## 13. 状態遷移と結果保護

| 操作 | 維持するもの | 無効化/破棄するもの | 次状態 |
|---|---|---|---|
| 構造、枝順、長さ、幅、用紙、格子、対称、配置を変更 | 変更前CPを閲覧用に保持 | 現入力に対する配置以降の保証 | `stale` |
| 名前、ラベル、表示色だけを変更 | CP、経路、検証結果 | なし | 状態維持 |
| 方式を変更 | ツリーと属性 | 配置、CP、M/V override、経路、積層、検証 | `none` |
| M/Vを編集 | CPと編集内容 | 経路、積層、検証 | `unverified` |
| FOLDを読込 | CP、再構築面 | 既存プロジェクトとは別に開く | `unverified` |
| 生成・検証が成功 | 新CP、経路、積層、検証詳細 | 同revision vectorの旧結果 | `verified` |
| 検証失敗後に下書きを採用 | 下書きCP | 保証表示 | `unverified` |
| 解なし根拠を確認 | 入力と根拠 | 現revision vectorのCP | `no-solution-certified` |
| 対応範囲判定に不合格 | 入力と理由 | ジョブ開始 | `unsupported` |
| ジョブをキャンセル/古い結果が到着 | 直前の全データ | 途中結果 | 状態維持 |

## 14. アクセシビリティ受け入れ基準

- 通常UI、フォーム、ダイアログ、通知、ツリー/CP編集はWCAG 2.2 AAを目標とする。
- 全入力に永続ラベル、エラーと説明の関連付け、可視フォーカス、論理的フォーカス順を持たせる。
- 状態と山谷は色だけで区別しない。高コントラスト時にもアイコン、文字、線種が残る。
- ツリーとCPの作成、選択、数値編集、削除、M/V変更、ズームをキーボードだけで実行できる。
- 進捗通知は過剰に読み上げず、工程変更と完了/失敗をARIA liveで通知する。
- 3D操作の既知制限をヘルプに明記し、同じ検証結果をテキストで取得可能にする。
