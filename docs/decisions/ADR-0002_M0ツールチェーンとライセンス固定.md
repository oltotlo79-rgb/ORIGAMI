# ADR-0002: M0ツールチェーンとライセンス適合版の固定

- 日付: 2026-07-14
- ステータス: Accepted
- 決定者: 実装担当（既存の要件・許可SPDX範囲内）
- 対象: M0開発基盤
- 上位要件: `../01_要件定義書_v1.1.md`、`../04_開発計画_マイルストーン.md`

## コンテキスト

M0は、production/developmentの直接・推移依存を`MIT`、`0BSD`、`BSD-2-Clause`、`BSD-3-Clause`、`Apache-2.0`、`ISC`だけに限定する。初期候補の最新版をそのまま組み合わせると、次の許可外依存が入った。

- Vite 8 → `lightningcss`の`MPL-2.0`
- Babelを使うReact plugin／React Hooks plugin → `caniuse-lite`の`CC-BY-4.0`
- 新しいtypescript-eslint → `minimatch`の`BlueOak-1.0.0`
- ESLintのlegacy config互換層 → `js-yaml` → `argparse`の`Python-2.0`

許可リストを暗黙に拡張することは要件違反である。一方、脆弱性のある旧ツール群へ全体を後退させることもM0基盤として不適切である。

## 決定

次の直接依存を固定する。

| 用途 | package | 固定版 | SPDX |
|---|---|---:|---|
| build | `vite` | 7.3.6 | MIT |
| React transform | `@vitejs/plugin-react-swc` | 4.2.2 | MIT |
| language | `typescript` | 5.9.3 | Apache-2.0 |
| lint | `eslint` / `@eslint/js` | 9.39.5 | MIT |
| TypeScript lint | `typescript-eslint` | 8.46.4 | MIT |
| Hooks lint | `eslint-plugin-react-hooks` | 5.2.0 | MIT |
| JSON Schema test | `ajv` | 8.20.0 | MIT |

`eslint-plugin-react-hooks`はBabel依存を持たない5.2.0に固定する。Viteは8ではなく、`lightningcss`を推移依存に持たない7.3.6を使う。TypeScript 6は対応typescript-eslintがBlueOak版`minimatch`を要求するため、5.9.3を使う。

`@vitejs/plugin-react-swc`は通常経路として維持する。ただしOSのApplication Control等がSWC native bindingを読み込めない場合、Vite設定はそのbinding failureだけを検出し、追加依存なしのVite標準JSX変換へfallbackする。fallback中はReact Fast Refreshを無効とするが、production build、CSP/network検査、および3 browser E2Eを同じscriptで継続する。Babel pluginを代替導入して許可外の推移ライセンスを増やしてはならない。

M0Fのchecked-in schemaは`ajv/dist/2020`のstrict modeでコンパイルし、実fixtureをDraft 2020-12として検証する。runtime parserだけ、またはschemaのID確認だけでは検出できない両者の乖離をCIで拒否する。

`@eslint/eslintrc`が内部で使う`js-yaml`のCLI専用依存については、npm overrideで`argparse@1.0.10`（MIT）を固定する。OriDesignは`js-yaml` CLIを実行せず、ESLintのflat configだけを使う。overrideにより`js-yaml` CLIの一部操作が壊れるため、そのCLIを開発手順へ追加してはならない。ESLint、型付きルール、React Hooksルール、`npm ci`、license検査をCIで実行して、この限定用途を継続検証する。

依存更新時は、lockfileを再生成して次を全て通す。

1. `npm run licenses:check`
2. `npm audit --audit-level=high`
3. `npm run lint`
4. `npm run typecheck`
5. `npm test`
6. `npm run test:e2e`

## 検討した代替案

### 許可SPDXを追加する

却下。ライセンス範囲の変更には上位要件とユーザー承認が必要であり、M0実装判断だけでは変更できない。

### Vite 8とTypeScript 6を使い、複数overrideで古い推移依存へ戻す

却下。Vite 8ではMPL依存が方式上必須であり、TypeScript 6ではさらにsemver範囲外の`minimatch` overrideが必要になる。非公式overrideを増やさない。

### ESLint 7以前へ下げる

却下。非サポート版と既知脆弱性を含む依存へ後退し、M0の長期基盤に適さない。

## 帰結

- 承認済み6種類以外のSPDXを追加せず、全依存を自動監査できる。
- `npm audit`で既知脆弱性0件のlockfileを維持する。
- 最新majorを無条件採用できないため、依存更新はlicense inventoryと実動作試験を伴う。
- SWC native bindingが許可されない開発環境ではFast Refreshを利用できないが、buildと検証はVite標準変換で継続できる。
- `argparse` overrideは限定的な技術的負債である。ESLintから当該依存が外れるか、上位要件で別ライセンスが承認された時点で解消を再検討する。
