# ADR-0003: M0F 座標・符号・面法線・積層規約 v1

- 日付: 2026-07-14
- ステータス: Accepted for M0F experiments
- 対象: M0F-0A
- 規約ID: `oridesign-m0f-conventions-v1`

## 背景

保存座標は左上原点、x右、y下である一方、剛体変換、面法線、FOLDの`edges_faces`、M/V、積層は右手系で一意に解釈する必要がある。変換境界を固定しないと、担当者やrendererごとに面法線、left/right、M/V、above/belowが反転し、後続の経路・CCD証跡を比較できない。

FOLD 1.2はfold angleをvalley正、mountain負と定義し、向き付き面法線と`edges_faces`のleft/rightを規定している。OriDesign固有の保存軸とこの規約を接続する必要がある。

## 決定

1. 保存紙面座標は従来どおり左上原点、x右、y下、短辺`1`とする。
2. 検証worldへ`(X, Y, Z) = (x, -y, z)`で写し、world `+Z`をeditor表面を見る観察者方向とする。
3. canonical faceはworld `+Z`から見て反時計回りとする。保存xyのsigned areaは負になる。
4. oriented edgeの`leftFaceId`/`rightFaceId`はworldで定義する。保存xyのorientation signではleftが負、rightが正になる。
5. fold angleはradianで保持してもFOLDと同じ符号を使い、正=`V`、負=`M`とする。
6. `aboveFaceId -> belowFaceId`を積層有向辺とし、`above`はworld `+Z`側を意味する。完全平坦時は距離でなくsymbolic orderであり、グラフ循環を禁止する。
7. M0F reference判定は有限binary64をexact dyadicへ復号し、orientationと面積符号をBigIntで決める。距離epsilonで符号を変えない。
8. 規約vectorのcanonicalizationはstable ID順とface ringの巡回回転だけを許し、ring反転とedge endpoint反転を行わない。

FOLDの符号定義は一次資料の[FOLD Specification 1.2](https://edemaine.github.io/fold/doc/spec.html)へ固定する。

## 却下した案

### 保存値そのものをy上へ変更する

却下。既存の要件・UI座標と逆になり、画面とfile境界の全箇所で変換漏れを起こしやすい。右手系への変換を検証境界で明示する方が追跡可能である。

### 画面上の見た目からM/Vと上下を決める

却下。camera、紙の表裏表示、renderer offsetで意味が変わり、独立checkerが再現できない。

### 単一epsilonでほぼゼロを共線・接触とする

却下。実際の交差を接触へ変えるfalse negativeを生み得る。高速filterはexact fallbackの発火条件にだけ使う。

## 影響

- `m0f/model`のruntime parserと規約evaluator、`m0f/geometry`のreference predicate、2面golden vectorを同じ規約へ固定する。
- FOLD import/export、target/path/CCD schemaはこの規約IDを保持する。
- 規約変更はIDとschemaを版上げし、全fixture、golden、mutationを再実行する。
- 本ADRはM0F実験の解釈を固定するだけであり、折り可能性、M0F `GO`、製品実装開始を承認しない。
