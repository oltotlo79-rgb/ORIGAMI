# ADR-0004: M0F候補契約と方式別境界 v1

- 日付: 2026-07-14
- ステータス: Accepted for M0F experiments
- 対象: M0F-0B

## 背景

ツリー法、本来の格子型箱ひだ、FOLD由来CPの検証は、入力条件も構成法も
終了根拠も異なる。一つの汎用`constraints`や一つの入力型へまとめると、
ある方式で測定した境界を別方式へ誤適用できる。また、試験用の経路候補が
構造検査を通っただけで`verified`と解釈される余地をなくす必要がある。

## 決定

1. SupportProfile候補は、ツリー法生成、箱ひだ生成、FOLD検証の3方式を
   判別可能な閉じた型として定義する。
2. 候補profileの値は候補列だけとし、`selected`、profile hash、evidenceは
   M0F測定完了までnull/pendingに固定する。
3. 試験artifactは入力、CreaseMesh、Target、候補経路、重なり、接触、積層を
   一つのcandidate contractで結ぶ。ただし`contractStatus: candidate`、
   `scientificClaim: false`を必須とし、検証結果欄を設けない。
4. JSON Schemaは全objectを閉じる。参照整合、木の閉路、面辺incidence、
   `[0,1]`被覆、積層DAGなどの横断不変条件はruntime parserでfail-closedに
   検査する。
5. ツリー法/箱ひだ入力は`uniaxial-tree-base`、FOLD入力は
   `flat-folded-cp`だけを対象とし、方式とTargetの混在を拒否する。
6. bounded interpolationとpiecewise polynomialは比較実験用候補に限る。
   M0Fで単一方式と証跡形式を選定するまで、どちらもPathCertificateではない。

## 却下した案

### 全方式を自由形式のprofileへまとめる

却下。未知constraint、方式固有値の混入、境界の誤用を機械的に防げない。

### candidate contractへ`verified: false`を持たせる

却下。後からbooleanを書き換えるだけで検証済みを装える形になる。科学的結果は
独立reference verifierが受理する別versionのartifact bundleでのみ表現する。

### 経路表現をM0F-0Bで即時固定する

却下。区間証明、閉路拘束、CCD bounds、ブラウザ性能の比較結果がまだない。

## 影響

- 製品solverやUIはまだ実装しない。
- 現在のschema/parser合格は科学的保証ではなく、M0F実験入力としての適格性だけを
  意味する。
- 三角形ID、region別目標積層、typed evidence、semantic hash、独立checkerを
  後続M0Fで固定し、新しいcertificate schemaへ版上げする。
