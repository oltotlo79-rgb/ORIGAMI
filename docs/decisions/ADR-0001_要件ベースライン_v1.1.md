# ADR-0001: OriDesign v1.1 要件ベースライン

- 日付: 2026-07-14
- ステータス: Accepted
- 決定者: プロダクトオーナー
- 対象: OriDesign v1
- 規範文書: `../01_要件定義書_v1.1.md`

## コンテキスト

初版要件には、次の不一致または未決定事項があった。

- 「前川・川崎チェックと同時折り 3D」と「実際に折れることの保証」が同一視されていた。
- 自己交差の警告要件に対し、実際の面衝突を検出する設計がなかった。
- 箱ひだがツリー法の格子配置差として扱われ、本来の格子型箱ひだに必要な方式固有構成がなかった。
- 任意比率の長方形に対し単一格子数 `N` を使い、正方形セルを維持できなかった。
- 葉 20 枚、3 秒/30 秒、保証判定の強制終了条件が競合していた。
- FOLD 面再構築、積層順序、連続折り経路、revision、複数タブ、自動復元等が要件化されていなかった。
- 無料配布の方法、公開ライセンス、対応 OS/ブラウザ、v1 の範囲が曖昧だった。

プロダクトオーナーとの確認により、候補 CP の提示ではなく、公開した対応範囲内で数値モデル上の連続折り経路を保証するプロダクトとすることが選択された。同時に、任意の折り紙や現実の紙まで保証対象に広げず、検証可能な対応プロファイルを先に確立する必要が生じた。

## 決定

### 1. v1.1 を規範ベースラインとする

`../01_要件定義書_v1.1.md` を v1 の要件ベースラインとする。v1.0 は履歴として残す。既存の技術設計、画面仕様、開発計画、開発規約と矛盾する場合は v1.1 を優先し、後続文書を改訂する。

### 2. 保証は「厚さゼロ数値モデル上の連続経路」とする

- 保証対象は、葉 2～20 枚かつ M0F で公開する方式別対応プロファイル内のツリーとする。
- 平面から目的の平坦 uniaxial base まで、連続折り経路、面の非貫通、接触時の積層順序を数値検証する。
- 接触と最終状態の重なりは許可し、面の貫通は禁止する。
- 前川・川崎の局所チェックだけでは検証済みにしない。
- これは数学的形式証明でも、有限厚の実物紙や人間による可折性の保証でもない。
- 保証対象入力は、利用者がキャンセルしない限り「検証済み」または根拠付き「解なし」へ確定させる。タイムアウトや数値発散を解なしと呼ばない。
- 対応プロファイル外は「対応範囲外」、インポート直後や手動編集後は「未検証」とし、状態を混同しない。

### 3. M0F を機能実装前のブロッキングゲートとする

一般 UI の実装より前に、本来の箱ひだ、葉 20 枚までの対応プロファイル、連続衝突判定、折り経路生成、積層順序、FOLD 面構築を試作する。

M0F で次を確定する。

- 対応可能な次数、ツリー形状、幅・長さ・用紙比率の数値範囲
- 箱ひだ `Nx`/`Ny` 上限と構成規則
- 距離、角度、接触、衝突、頂点統合等の許容誤差
- 解なしの決定手続と判定根拠
- ファイルサイズ・頂点・辺・面の上限
- 性能基準 CPU 型番、ブラウザ、測定手順と実測値

M0F が No-Go の場合は機能実装へ進まず、要件変更を再承認する。保証を候補表示へ暗黙に格下げしない。

### 4. 箱ひだを独立した方式として扱う

箱ひだは、ツリー法の葉配置を格子に制限しただけの実装にしない。格子型箱ひだ固有の軸線、river、分子、折り線を生成する。長方形では正方形セルを保ち、`Nx` と `Ny` を分離する。

実数の枝長・幅は格子値へ近似し、1% 以内を合格とする。元値、格子値、誤差を表示し、条件を満たせない場合は細かい格子を提案するか、対応格子範囲内で解なしを返す。

### 5. 幅と最終形を保証入力に含める

幅 `w` は末端枝だけでなく内部枝にも設定し、完成した枝/river の横幅を表す。分岐部や用紙へ収まらなければ解なしにできる。最終形は枝構造、長さ比、幅を反映した平坦な uniaxial base とする。

### 6. 検証状態を編集・保存・出力の全域で追跡する

- M/V 手動編集、幾何・方式・配置変更は検証済み状態を解除する。
- `design`、`layout`、`cp` の revision vector と、各成果物が依存する成分・生成元 hash を保存し、古い Worker 結果を適用しない。名称、ラベル、表示設定では vector を進めない。
- 未検証または stale な CP も警告付きで表示、編集、3D 読込、再出力できる。
- 検証済みの 3D スライダーは一律折り率ではなく、保存された折り経路上の時刻を示す。
- キャンセルや失敗で最後の検証済み結果を上書きしない。

### 7. FOLD は編集可能だが初期状態を未検証とする

FOLD 1.1/1.2 の top-level key frame にある、有効フレームが厳密に 1 件の 2D 平面 CP を正式対応し、`faces_vertices` がなければ面を再構築する。CP 表示、M/V 編集、再出力、3D ビューへの読込を許可する。既定出力は現行仕様の FOLD 1.2 とし、custom property 規則に従う。1 件以上の `file_frames`、初期 3D、非平面、非多様体、assignment `C/J` は理由を示して非対応とする。FOLD 内の検証メタデータは信用せず、読込直後は常に未検証とする。FOLD 由来データはツリーなしの FOLD プロジェクトとして保存できる。検証時はツリーを推定せず、M/V と整合する完全平坦目標と専用 SupportProfile を M0F で確定する。

### 8. v1 は無料公開 Web/PWA とする

- 公開リポジトリと静的ホスティングで Web/PWA を配布する。
- Windows 10/11 の最新版 Chrome、Edge、Firefox を対象とし、Safari とモバイル最適化は外す。
- PWA install UI はブラウザが提供する場合に対応し、非提供ブラウザでは公開 URL を正式な利用経路とする。
- 常時インターネット接続を前提とし、完全オフライン配布は要求しない。
- v1 は M0～M9 相当 + PWA とし、段折り（旧 M10）と Tauri 版は後続へ送る。
- PWA は作業中に自動更新せず、保存後の利用者操作で更新する。

### 9. ローカル保存の安全性を製品要件とする

UUID、編集停止 2 秒後の IndexedDB 保存、最近 10 プロジェクト、選択式復元、ファイル保存後の復旧データ保持、複数タブの読取り専用化と編集引継ぎ、既定 100/最大 500 の Undo/Redo、revision/stale 管理を必須とする。

ファイルは実行時スキーマ検証し、上書き保存に非対応のブラウザではダウンロードへフォールバックする。保存成功時だけ dirty を解除する。

### 10. 公開・プライバシー・ライセンス方針を固定する

- 本体を MIT で公開し、改変、再配布、商用利用を許可する。
- 初期許可 SPDX ID は `MIT`、`0BSD`、`BSD-2-Clause`、`BSD-3-Clause`、`Apache-2.0`、`ISC` とし、CI で検査する。
- GPL コードを依存、コピー、翻案、参照移植せず、論文・公開数理記述から独立実装する。
- 静的アセット取得と PWA 更新確認以外の外部通信を行わず、解析、広告、クラッシュ送信、外部 API、外部フォント、CDN を使わない。

## 検討した代替案

### 局所定理と同時折り 3D を「折れる」の根拠にする

却下。前川・川崎の局所条件と離散的な同時折り表示だけでは、初期状態から目的状態までの自己貫通しない連続経路を示せない。

### すべての任意ツリーを無条件に保証する

却下。完成時期を見積もれず、解なしとの区別も定義できない。葉 20 枚を上限とし、M0F で検証済みの対応プロファイルを公開する。ただしプロファイル内では候補扱いに逃げず、保証契約を満たす。

### 数学的証明または現実の紙での可折性を保証する

却下。v1 は厚さゼロの数値モデルと公開許容誤差に限定する。物性、摩擦、手指の可達性、形式証明はスコープ外とする。

### 箱ひだをツリー法の格子スナップとして共通実装する

却下。本来の格子型箱ひだという利用者期待を満たさない。共通の幾何ユーティリティは再利用できるが、生成工程は方式別に持つ。

### デスクトップアプリを先に配布する

却下。無料で他 PC に配布しやすい公開 Web/PWA を v1 とする。Tauri は将来候補に残す。

### 3 秒/30 秒で計算を強制終了する

却下。保証対象入力を「解なし」へ誤分類する危険がある。値は配置の努力目標とし、保証処理は進捗表示と利用者キャンセルを採用する。

## 帰結

### 利点

- 「検証済み」が何を保証し、何を保証しないかを一貫して説明できる。
- 未検証ドラフトや外部 FOLD を排除せず、安全に編集・再検証できる。
- 箱ひだの利用者期待と長方形格子のデータモデルが一致する。
- 無料配布、データ非送信、公開ライセンスという配布方針が明確になる。
- revision、復元、複数タブ、更新制御により長時間計算中のデータ消失を防げる。

### コストとリスク

- 連続衝突判定、経路生成、積層順序、正しい解なし判定は研究開発性が高い。
- M0F が No-Go となり、要件の再審議が必要になる可能性がある。
- 固定タイムアウトを設けないため、難しい入力では計算時間を事前確約できない。
- 本来の箱ひだとツリー法は生成工程を分ける必要があり、共通コアだけの計画より工数が増える。
- 未検証データの利用を許すため、画面と出力で状態を誤認させない設計が必要になる。

## 遵守確認

次の変更は ADR の更新または新規 ADR と、プロダクトオーナーの承認を必須とする。

- 葉 20 枚上限の引下げ、または保証対象から試験機能への降格
- 連続衝突判定を離散サンプルだけの判定へ変更
- 解なしにタイムアウト・数値発散を含める変更
- 本来の箱ひだを格子スナップ方式へ簡略化する変更
- 未検証データを検証済みと同じ表示・出力にする変更
- v1 に段折り、物理紙保証、Tauri 配布を追加する変更
- 対応環境、通信、公開、ライセンス方針を緩和する変更
