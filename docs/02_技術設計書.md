# OriDesign 技術設計書 v1.1

作成日: 2026-07-14  
ステータス: 実装準備版（本番機能の実装開始には M0F の Go 判定が必要）

## 1. 設計原則と保証境界

1. 本アプリが「検証済み」と表示する展開図は、M0F で確定する対応範囲内において、厚さゼロ・面剛体・折り線ヒンジモデル上の連続折り経路を持つ。
2. 経路の全区間で面の貫通を認めない。接触、最終状態での面の重なり、順序付けされた積層は認める。
3. 保証は規定の数値許容誤差内で行う工学的保証であり、現実の紙の厚さ、弾性、加工誤差、人間による折りやすさは保証しない。
4. 前川・川崎チェック、エネルギー収束、離散フレームの見た目だけでは「検証済み」にしない。
5. 自動生成、FOLD 読込、M/V 手動変更のいずれも、入力ハッシュと検証証跡が一致した場合だけ「検証済み」にする。
6. 自動生成方式は、ツリー法と本来の uniaxial box pleating を別パイプラインとして持つ。共通化するのはデータ構造、幾何基盤、メッシュ化、検証、表示、出力である。
7. 重い処理は専用 Web Worker で実行する。強制タイムアウトは設けず、進捗表示とユーザーキャンセルに対応する。

## 2. 実装開始ゲート

本番機能へ進む前に `05_実現性検証仕様書.md` の M0F を完了し、次を確定する。

- `SupportProfile`：葉2〜20を前提とした対応可能な次数、形状、幅、格子、縮退条件
- ツリー法および箱ひだの、対応範囲内で完備な候補生成手順
- 連続折り経路生成方式と停止条件
- false negative（貫通の見逃し）を生じない連続衝突判定方式
- 接触と貫通を分ける数値許容誤差 `ToleranceProfile`
- 箱ひだ格子数の上限、入力近似規則、1%誤差条件
- 葉20の固定 fixture におけるメモリ・実行時間
- FOLD 面構築と積層順序の表現方法

M0F が No-Go の場合、対応範囲または保証内容を ADR で変更しない限り本番実装へ進まない。

## 5. 座標・単位・許容誤差

### 5.1 内部座標

- 用紙の短辺を `1` とする無次元座標を用いる。
- 原点は用紙左上、x は右、y は下。幾何計算で必要な右手系変換は境界で明示する。
- 検証用右手系worldへの変換は `(X, Y, Z) = (x, -y, z)` とし、world `+Z` をeditor表面側とする。
- canonical faceはworld `+Z`から見て反時計回り、すなわち保存xyのsigned areaが負となる向きにする。edgeのleft/rightもworldで定義する。
- 枝長と幅は同じ無次元比率単位で入力し、配置スケール `s` で紙上長へ変換する。
- PDF は指定用紙へ余白内で最大フィットする。実寸保証は行わない。
- FOLD は `frame_unit: "unit"`、SVG は `viewBox` に正規化座標を出力する。

### 5.2 ToleranceProfile

単一の `EPS` で全判定を行わず、M0F で次を別々に確定する。

```ts
interface ToleranceProfile {
  id: string;
  schemaVersion: 1;
  profileHash: string;
  coordMergeAbs: number;
  orientPredicate: {
    fastFilterArea: number;
    fallback: 'adaptive-exact';
  };
  minFaceArea: number;
  rigidDistanceAbs: number;
  rigidDistanceRel: number;
  hingeCoincidenceAbs: number;
  loopClosureAbs: number;
  loopClosureAngle: number;
  pathContinuityAbs: number;
  pathContinuityAngle: number;
  goalFlatnessAbs: number;
  goalFoldAngle: number;
  treeLengthRel: number;
  treeWidthRel: number;
  gridQuantizationRel: number; // v1 は 0.01 以下
  contactClassification: {
    method: 'oriented-side-history';
    ambiguous: 'adaptive-exact-or-fail';
  };
  timeCoverage: {
    maxGap: number;
    endpointOverlap: number;
  };
  displayEndpointOffset: number;
}
```

検証証跡は使用した profile ID と全値を保持する。値を変更した場合、既存の検証結果は stale とする。

### 5.3 canonicalization と hash

- hash は Web Crypto の SHA-256 を使用し、`canonicalizationVersion` とともに保存する。
- JSON object key、有限数の表現、stable ID による map/集合順、FOLD parallel array の対応を固定した `oridesign-canonical-v1` を M0F で fixture 化する。
- `inputHash` は意味的入力だけ、`foldHash` は `oridesign:verification` を除く正規化済み FOLD、`certificateHash` は自身の hash field を除く certificate から計算する。
- metadata の表示順、timestamp、ファイル名は幾何・証跡 hash に含めない。配列順に意味がある `faces_vertices` 等は勝手に sort せず、規定の canonical orientation/rotation だけを適用する。
- hash は真正性や作成者を証明する署名ではなく、同じデータを結ぶ整合性キーである。最終判定は reference checker が内容を再計算して行う。

ツリー画面の位置は枝の巡回順を定める。内部ノードで incident edge の角度順が変わったときだけ `rotation` を更新し、revision vector の `design` を進める。位置を動かしても保存済み巡回順が変わらない場合は表示上の変更として扱う。

### 6.4 派生状態と stale

```ts
type RevisionDependency = Readonly<Partial<RevisionVector>>;

interface ArtifactStamp {
  requiredRevisions: RevisionDependency;
  inputHash: string;
  supportProfileId?: string;
  supportProfileHash?: string;
  toleranceProfileId?: string;
  toleranceProfileHash?: string;
  verifierVersion?: string;
  canonicalizationVersion: string;
}

interface DerivedArtifact<T> {
  stamp: ArtifactStamp;
  value: T;
}

interface DerivedDesignState {
  layout?: DerivedArtifact<LeafLayout>;
  creaseMesh?: DerivedArtifact<CreaseMesh>;
  fold?: DerivedArtifact<FoldFile>;
  target?: DerivedArtifact<TargetState>;
  pathCertificate?: DerivedArtifact<PathCertificate>;
  outcome?: DerivedArtifact<PersistedSolveOutcome>;
  candidateDiagnostics?: DerivedArtifact<CandidateDiagnostics>;
}

interface DerivedFoldState {
  creaseMesh?: DerivedArtifact<CreaseMesh>;
  target?: DerivedArtifact<TargetState>;
  pathCertificate?: DerivedArtifact<PathCertificate>;
  outcome?: DerivedArtifact<PersistedSolveOutcome>;
  candidateDiagnostics?: DerivedArtifact<CandidateDiagnostics>;
}

interface LeafLayout {
  scale: number;
  source: 'automatic' | 'manual';
  positions: Record<Id, { x: number; y: number }>;
}

interface CreaseMesh {
  vertices: Array<{ id: Id; x: number; y: number }>;
  edges: CreaseEdge[];
  faces: Array<{ id: Id; vertices: Id[] }>;
  meshJoinEdges: MeshJoinEdge[];
}

interface MeshJoinEdge {
  id: Id;
  vertices: [Id, Id];
  purpose: 'triangulation' | 'overlap-subdivision';
  incidentFaceIds: Id[];
}

interface CandidateDiagnostics {
  phase: string;
  messageKeys: string[];
  targetIds: Id[];
  metrics: Record<string, number>;
}
```

依存関係ごとに revision を進める。

- 構造、長さ、幅、用紙、格子、対称の変更：`design` を進め、layout／CP／verification を stale にする。
- 方式変更：`design` を進めるが stale 表示にはせず、ツリー以外の layout／CP／M/V override／target／path／outcome を現在状態から破棄して `none` にする。Undo 前の履歴内参照だけは保持する。
- 自動または手動配置の変更：`layout` を進め、CP／verification を stale にする。
- crease geometry、assignment、M/V の変更：`cp` を進め、verification を stale にする。M/V 変更を反映した CP 自体は current とする。
- 名前、ラベル、表示設定：いずれの revision も進めない。

artifact の `requiredRevisions` に記録した成分、意味的入力 hash、または必要な ToleranceProfile のいずれかが現在値と一致しなければ stale である。layout は原則 `design`、CP は `design` と `layout`、検証結果は `design`、`layout`、`cp` に依存する。FOLD プロジェクトは `cp` を主成分とする。正確な依存表は runtime schema と unit test で固定する。

`inputHash` は表示名、ラベル、timestamp、端末設定、派生状態を除外した canonical semantic input から計算する。これにより、検証意味を変えない metadata 編集で証跡を不必要に stale にしない。

古い派生結果は表示できるが、検証済み表示と新規の保証付き出力には使用しない。

### 6.5 stable crease identity

```ts
interface CreaseEdge {
  id: Id;
  vertices: [Id, Id];
  assignment: 'M' | 'V' | 'B' | 'F' | 'U';
  role: 'boundary' | 'axial' | 'axialN' | 'ridge' | 'hinge' | 'gusset' | 'pleat' | 'flat';
  sourceTreeEdgeIds: Id[];
  generationKey: string;
}
```

再生成時は `generationKey` が一致した折り線に限り M/V override を継承する。一致しない override は破棄し、件数を通知する。手動 M/V 編集は verification を即時 stale にする。

### 6.6 formatVersion 1 からの migration

v1 ファイルをそのまま型 assertion せず、runtime schema で旧形式を確認してから次の純粋関数で変換する。

```ts
interface MigrationResult {
  project: ProjectFileV2;
  warningKeys: string[];
}

type MigrateProject = (legacy: unknown) => Result<MigrationResult, DomainError>;
```

- UUID、timestamp、RevisionVector を新規付与する。
- 旧 `TreeEdge` に label がなければ空文字を設定する。
- 旧 layout／CP は生成元 revision と証跡を持たないため、保持しても必ず stale／unverified とする。
- 旧 symmetry が boolean だけで mirror 対応を一意に復元できない場合、座標から推測して自動確定しない。対称を OFF にし、警告と再設定導線を出す。
- 旧 FOLD 1.1 は対応 subset として正規化するが、検証済み metadata は信用しない。
- 未知の将来 `formatVersion` は内容を上書きせず、非対応 version として読込を中止する。

## 7. 入力状態とソルバー結果契約

### 7.1 入力分類

1. `InvalidInput`：JSON、参照、有限数、ツリー構造、正値条件などに違反
2. `UnsupportedInput`：`SupportProfile` の葉数・次数・縮退・幅・格子条件外
3. `SupportedInput`：M0F で保証可能と確定した範囲内

構造的に正しいだけでは `SupportedInput` とは限らない。UI は理由と制限値を表示する。

生成と FOLD 検証は入力の意味が異なるため、別 profile とする。M0F は次の provisional schema を具体化し、凍結 JSON Schema から TypeScript 型を生成する。`constraints` の汎用表現は M0F 中だけ許し、`GO` 時には判別可能な具体型へ置換する。

```ts
interface SupportProfileBase {
  id: string;
  schemaVersion: 1;
  profileHash: string;
  constraintsSchemaId: string;
  constraints: Record<string, JsonValue>; // M0F GO時に具体型へ置換
}

interface GenerationSupportProfile extends SupportProfileBase {
  kind: 'design-generation';
  method: 'treeMethod' | 'boxPleating';
  leaves: { min: 2; max: 20 };
}

interface FoldVerificationSupportProfile extends SupportProfileBase {
  kind: 'fold-verification';
  assignments: Array<'M' | 'V' | 'B' | 'F' | 'U'>;
  maxVertices: number;
  maxEdges: number;
  maxFaces: number;
}

type SupportProfile = GenerationSupportProfile | FoldVerificationSupportProfile;

type SupportDecision =
  | { supported: true; profileId: string; profileHash: string; normalizedInputHash: string }
  | { supported: false; profileId: string; profileHash: string; violations: SupportViolation[] };

type JsonValue =
  | null
  | boolean
  | number
  | string
  | JsonValue[]
  | { [key: string]: JsonValue };
```

### 7.2 完了結果

```ts
type SolveResult =
  | { kind: 'verified'; solution: VerifiedSolution }
  | { kind: 'no-solution-certified'; certificate: NoSolutionCertificate }
  | { kind: 'cancelled' }
  | { kind: 'unsupported'; violations: SupportViolation[] }
  | { kind: 'invalid'; violations: InputViolation[] }
  | { kind: 'internal-error'; diagnosticId: string };

interface VerifiedSolution {

  fold: FoldFile;
  target: TargetState;

  supportProfileHash: string;
  toleranceProfileId: string;
  toleranceProfileHash: string;
  canonicalizationVersion: string;
  inputHash: string;

  certificateHash: string;
}

interface WitnessBase {
  schemaVersion: 1;
  completenessArgumentId: string;
}

interface GridExhaustionWitness extends WitnessBase {
  kind: 'grid-exhaustion';
  nxRange: [number, number];
  nyRange: [number, number];
  rejectedCandidates: Array<{ nx: number; ny: number; reasonCode: string }>;
}

interface PaperDiameterWitness extends WitnessBase {
  kind: 'paper-diameter';
  required: number;
  available: number;
}

interface WidthCapacityWitness extends WitnessBase {
  kind: 'width-capacity';
  targetEdgeIds: Id[];
  required: number;
  available: number;
}

  kind: 'construction-precondition';
  ruleId: string;
  targetIds: Id[];
  actual: JsonValue;
  required: JsonValue;
}

interface FiniteSearchExhaustionWitness extends WitnessBase {
  kind: 'finite-search-exhaustion';
  searchSchemaId: string;
  domainHash: string;
  checkedStateCount: number;
  rejectionSummary: Record<string, number>;
}

type NoSolutionWitness =
  | GridExhaustionWitness
  | PaperDiameterWitness
  | WidthCapacityWitness
  | ConstructionPreconditionWitness
  | FiniteSearchExhaustionWitness;

type PersistedSolveOutcome =
  | { kind: 'verified'; solution: VerifiedSolution }
  | { kind: 'no-solution-certified'; certificate: NoSolutionCertificate };

interface SupportViolation {
  code: string;
  targetIds: Id[];
  actual: JsonValue;
  allowed: JsonValue;
  remediationKey?: string;
}

  code: string;
  targetIds: Id[];
  messageKey: string;

```

対応範囲内の処理にタイムアウト結果は設けない。ユーザーがキャンセルしない限り、`verified` または `no-solution-certified` へ停止できる方式だけを M0F で採用する。候補は生成できたが証跡を完成できない場合、候補を未検証 draft として保持できるが、solve の成功には数えない。

`NoSolutionCertificate` の意味と検証方法は M0F で方式別に定義する。単なる探索打切りや局所最適化失敗を「解なし」としてはならない。

`cancelled`、`internal-error`、進行中状態はジョブの一時状態であり、直前の `PersistedSolveOutcome` を上書きしない。`unsupported` と `invalid` は現在入力から同期的に再計算する。保存対象の正常終端は `verified` と `no-solution-certified` だけである。

## 8. ツリー法パイプライン

```text
EmbeddedTree + Paper
  -> SupportProfile 検査
  -> circle/river packing 候補生成
  -> path condition 検査と active set 確定
  -> Lang polygon / embedded subtree 構築
  -> universal molecule 候補生成
  -> planarize + half-edge 面構築
  -> M/V・積層候補
  -> 連続折り経路探索
  -> CCD + goal match
  -> VerifiedSolution
```

### 8.1 パス条件

葉 `i,j` について、配置 `u_i,u_j` とスケール `s` が次を満たすことを必要条件とする。

`distance(u_i, u_j) >= s * metricDistance(i, j)`

幅を含む `metricDistance` の厳密式、境界パス、内部 river の扱いは M0F で構成法と同時に固定する。旧仕様の「経路上の各枝へ単純に w を加算」は採用済み仕様ではない。

### 8.2 埋め込みと universal molecule

universal molecule の入力は多角形だけでなく、対応する埋め込み付き metric subtree を含む。marker、縮約パス、同時イベント、共線／近接頂点を型で明示し、イベント順を決定的にする。

近似最適化結果を固定の `1e-6*s` だけで active 判定しない。制約射影または active-set refinement を行い、採用した等号制約と残差を証跡へ保存する。

universal molecule から得た CP が連続剛体折り可能であるとは仮定せず、必ず第11章の検証へ通す。

## 9. 本来の箱ひだパイプライン

箱ひだはツリー法配置の格子量子化として実装しない。

```text
EmbeddedTree + Rectangle + GridSpec
  -> 実数長・幅・用紙比の格子近似レポート
  -> polygon/river packing
  -> axial / axial+N / ridge / hinge / gusset 構築
  -> planarize + half-edge 面構築
  -> M/V・積層候補
  -> 連続折り経路探索
  -> CCD + goal match
  -> VerifiedSolution
```

- セルは正方形とし `columns` と `rows` を別管理する。
- 実数長、幅、用紙比を格子値へ近似し、各項目の相対誤差を表示する。
- v1 の合格条件は各対象値の相対誤差 `<= 1%`。満たせなければ細かい格子を提案し、上限内に候補がなければ解なしまたは対応範囲外を、M0F で定めた証跡とともに返す。
- `N=4` 相当を許可する。最大 columns/rows と探索方法は M0F で固定する。
- axial+N を含む箱ひだ固有線を `role` として保持し、FOLD と SVG へ出力する。

## 10. CP の平面グラフ化

候補生成器が返す線分集合を、そのまま FOLD や 3D へ渡してはならない。次を一つのトランザクションとして実行する。

1. 用紙境界でクリップ
2. 全交点と T-junction で線分を分割
3. 許容誤差内の頂点を統合
4. 重複・ゼロ長辺を除去し、割当競合を検出
5. half-edge/DCEL を構築
6. 外部面を除外し、内部面を CCW で列挙
7. Euler 特性、辺の incident face 数、面積、向きを検査
8. 各単純面を三角形分割し、面積差・反転・T-junction がないことを検査
9. stable crease ID と元ツリー枝の対応を確定

失敗時は幾何診断を返し、検証へ進まない。

## 11. M/V・積層・連続折り経路

### 11.1 M/V

- fold angleの符号はFOLD 1.2に合わせ、正を`V`、負を`M`とする。保存座標からworldへの変換で符号を暗黙反転しない。
- 検証済み解の M/V は、折り経路と面の向き・積層順序から導出する。
- `axial=V` 等の固定ヒューリスティックは draft の初期候補に限る。
- 前川違反は候補 M/V の再探索対象とする。
- 川崎違反は角度の問題であり M/V 編集では直せない。候補幾何へ戻す。
- 前川・川崎の合格表示は「局所必要条件合格」と表記し、大域保証と区別する。

### 11.2 経路モデル

```ts
interface RigidTransform {

  translation: [number, number, number];
}

interface EvidenceRef {
  hash: string;

}

interface EvidenceBlob {
```

経路パラメータ `t=0..1` は時間位置であり、全折り線の一律な折り率ではない。各折り線は別の角度・速度で動ける。

`CreaseMotion` の採用 variant、次数・区間上限および各 evidence の具体 schema は M0F で一つに固定する。`GO` 時に未選択 variant や汎用 payload を製品契約へ残さない。

全ての `*ProofHash` / `*EvidenceHash` は `evidenceManifest` の `EvidenceRef.hash` に解決し、対応する `EvidenceBlob` 本体を `.orid.json`、snapshot、reference-checker fixture に同梱する。hash だけを保存して証跡本体を失う状態を schema で拒否する。

### 11.3 目標状態

```ts
interface TargetStateBase {
  schemaVersion: 1;
  faceTransforms: Record<Id, RigidTransform>;
  goalFaceOrders: Array<[Id, Id, -1 | 1]>;
  overlapRegionHashes: string[];
  targetHash: string;
}

interface UniaxialTargetState extends TargetStateBase {
  kind: 'uniaxial-tree-base';
  branchMeasurements: Array<{
    treeEdgeId: Id;
    axisEndpoints: [[number, number], [number, number]];
    effectiveLength: number;
    effectiveWidth: number;
  }>;
  rotation: Record<Id, Id[]>;
}

interface ImportedFoldTargetState extends TargetStateBase {
  kind: 'flat-folded-cp';
  assignmentPolicy: 'respect-mv-solve-u';
  resolvedInteriorAssignments: Record<Id, 'M' | 'V' | 'F'>;
}

type TargetState = UniaxialTargetState | ImportedFoldTargetState;
```

設計プロジェクトの目標は、入力ツリーのトポロジ、長さ比、幅を規定誤差内で反映する平坦な uniaxial base とする。FOLD プロジェクトの目標は、現在の CP 幾何を変えず、M/V を尊重し、`U` を明示解決した任意の無矛盾な完全平坦状態とする。FOLD 検証 profile の範囲内だけ決定保証を行い、範囲外でも候補表示・編集・出力は許可する。

数理検証は `t=1` の完全平坦状態を扱い、3D 描画は z-fighting 回避のため直前状態または積層オフセット表示を選べる。

### 11.4 連続衝突判定

- 非隣接三角形対の全経路区間を連続判定する。
- 隣接面でも共有ヒンジ線上の合法な接触だけを除外し、それ以外の領域での貫通は検査する。
- broad phase と conservative narrow phase を分離する。
- 離散サンプリングだけを合格根拠にしない。
- 接触状態では overlap region ごとの `LayerOrderEvent` が時間方向に無矛盾であり、終端の `goalFaceOrders` が非循環であることを確認する。
- M0F で確定した SupportProfile、ToleranceProfile、経路表現の範囲内で CCD の false negative がない保守性根拠を示し、adversarial fixture で検証する。

採用アルゴリズム、補間の上界、接触の退化処理は M0F で確定する。

## 12. FOLD 入出力

### 12.1 対応範囲

- raw JSON は `unknown` として受け、runtime schema 後に `FoldFile`（内部の正規化済み FOLD 1.2）へ変換する
- import: FOLD 1.1 / 1.2 の top-level key frame、2D、平面 CP。`file_frames` は不存在または空配列に限る
- `vertices_coords`、`edges_vertices`、`edges_assignment` を必須とする
- `faces_vertices` がなければ第10章で再構築する
- M/V/B/F/U を正式対応し、C/J、非平面、非多様体、3D 初期 frame、複数 frame は理由付き非対応とする
- 未知の custom field は、トポロジを変更しない round-trip では保持する。再平面化で対応が壊れる field は警告して除外する
- `.fold` の custom verification metadata は表示用の参考情報に限り、読込時の状態は必ず `unverified` とする。文字列、hash、外部 tool の署名だけで `verified` に昇格させない

### 12.2 export

FOLD 1.2 を既定出力とし、主要外部ツール向け互換 fixture で確認する。

```ts
interface FoldFile {
  file_spec: 1.2;
  file_creator: 'OriDesign';
  file_classes: string[];
  frame_attributes: string[];
  frame_unit: 'unit';

  edges_vertices: [number, number][];
  edges_assignment: ('M' | 'V' | 'B' | 'F' | 'U')[];
  faces_vertices: number[][];
  faceOrders?: [number, number, -1 | 0 | 1][];
  'edges_oridesign:id'?: string[];
  'edges_oridesign:role'?: CreaseEdge['role'][];
  'edges_oridesign:sourceTreeEdges'?: string[][];
  'oridesign:verification'?: FoldVerificationMetadata;
}

interface FoldVerificationMetadata {
  status: 'verified' | 'unverified' | 'stale';
  verifierVersion?: string;
  supportProfileId?: string;
  supportProfileHash?: string;
  toleranceProfileId?: string;
  toleranceProfileHash?: string;

```

parallel array は対象配列と同じ長さにする。未検証出力には status、input hash、verifier version、tolerance profile を付ける。検証済み出力でも、再 import は常に未検証から開始する。

`MeshJoinEdge` は剛体面の三角形分割または overlap region の細分に使う内部補助辺であり、物理的な折り線ではない。FOLD の `edges_*` へは出力せず、細分後の `faces_vertices` と `.orid.json` の証跡にだけ反映する。したがって外部 FOLD の `J` を v1 で拒否する方針と両立する。

`.orid.json` の検証済み状態を復元する場合は、runtime schema、revision vector、全 hash、SupportProfile/ToleranceProfile/verifier 版を照合し、保存済み証跡をローカル reference checker が受理した場合に限る。不一致または利用不能な旧 checker の結果は `stale` とする。

## 13. 3D 表示

- 3D は検証済み `PathCertificate.path` を再生する。スライダー 0〜100% は `t=0..1` に対応する。
- 一時停止、再生、逆再生、任意時刻シークを行える。
- 検証済み経路では CCD 証跡と同じ補間を用いる。表示補間だけを独自に平滑化しない。
- 未検証 CP は探索中の参考表示として明示し、「実際に折れる」と表示しない。
- 表面／裏面色、積層分離表示、接触箇所、衝突診断を表示できる。
- 必須性能は基準端末で三角形1,000枚・30fps、2,000枚は努力目標。検証計算の速度とは分離する。

## 14. Worker プロトコルと競合制御

```ts
type WorkerRequest =
  | { type: 'start'; jobId: string; revisions: RevisionVector; inputHash: string; task: WorkerTask }
  | { type: 'cancel'; jobId: string };

type WorkerResponse =
  | { type: 'progress'; jobId: string; revisions: RevisionVector; phase: string; completed: number; total?: number }
  | { type: 'draft'; jobId: string; revisions: RevisionVector; diagnostic: CandidateDiagnostics }
  | { type: 'done'; jobId: string; revisions: RevisionVector; result: SolveResult }
  | { type: 'error'; jobId: string; revisions: RevisionVector; error: SerializedSolverError };

type WorkerTask =
  | { kind: 'generate-and-verify'; project: DesignProject }
  | { kind: 'verify-fold'; project: FoldProject }
  | { kind: 'propose-grid'; project: DesignProject };

interface SerializedSolverError {
  code: string;
  phase: string;
  targetIds: string[];
  diagnosticId: string;
}
```

- 計算ごとに専用 Worker を作り、キャンセル時は `terminate()` して確実に停止する。
- UI は `jobId`、必要な revision vector 成分、hash が現在値と一致する完了結果だけを適用する。
- キャンセル、失敗、古いジョブの結果で既存の検証済みデータを上書きしない。
- SharedArrayBuffer や特殊な配信 header を必須にしない。
- 進捗単位と cancellation checkpoint は M0F で各アルゴリズムについて定義する。

## 15. 状態管理・Undo/Redo

- 入力状態、派生状態、端末設定を別 slice にする。
- 1ドラッグ、1セル確定、1コマンドを1履歴とする。
- ツリー、用紙、方式、格子、対称、配置、M/V、solver result 適用を履歴対象とする。
- 配色、履歴上限、3D表示設定は端末設定であり履歴対象外。
- 既定100件、設定可能上限500件。件数に加え M0F で定める byte budget を超えた場合は古い履歴から破棄する。
- 大きな CP／経路は content hash で `DerivedStateStore` に置き、履歴は参照を持つ。参照中 blob は破棄しない。
- 方式切替ではツリーだけを維持し、配置、CP、M/V、検証結果を破棄する。箱ひだへ切替時は格子候補を提案する。

## 16. 保存・復旧・複数タブ

- プロジェクト UUID をキーに、編集停止2秒後に IndexedDB へ原子的 snapshot を保存する。
- 最近使った10プロジェクトを保持する。起動時は自動復元せず、復元／破棄を選ばせる。
- ファイル保存後も recovery snapshot を残し、ユーザーが削除できる。
- BroadcastChannel と heartbeat で同一プロジェクトの編集タブを調整する。後から開いたタブは read-only とし、明示的 takeover を行える。
- takeover 時は旧タブを read-only にし、両方の未保存 snapshot がある場合は選択画面を出す。自動 merge はしない。
- 保存 API 対応環境では同一ファイルへ上書きし、非対応環境ではdownloadする。上書きAPI成功時、またはdownload後に利用者が「保存した」を明示確認した場合だけdirtyを解除する。download開始イベントだけでは解除しない。
- PWA 更新は作業中に自動適用せず、保存後の更新を促す。

## 17. runtime validation と安全性

- TypeScript 型だけに依存せず `.orid.json` と `.fold` を runtime schema で検証する。
- file bytes、配列数、文字列長、有限数、座標範囲、ID／index 参照、重複、面妥当性を検査する。
- 上限値は M0F の性能・メモリ測定後に固定する。
- ラベル、プロジェクト名、ファイル名を SVG／HTML／PDF へ出す際はエスケープする。
- 外部 API、analytics、広告、crash upload、外部 font、CDN を使用しない。
- CSP を設定し、ビルド成果物に外部 URL がないことを E2E で検査する。

## 18. アクセシビリティ

- 通常 UI と form は WCAG 2.2 AA を目標とする。
- 色だけで状態を伝えず、線種、アイコン、テキストを併用する。
- ツリーと CP の選択・編集に keyboard 代替を設ける。
- 3D の完全代替が困難な操作は既知制限として文書化し、数値の経路時刻と検証結果は DOM でも読めるようにする。

## 19. エラー処理

solver／verify は例外を公開 API の制御フローに使わず `Result<T, DomainError>` を返す。programming error は Worker 境界で捕捉し diagnostic ID として扱う。

```ts
interface DomainError {
  code: string;
  phase: string;
  targetIds: string[];
  messageKey: string;
  details?: Record<string, string | number | boolean>;
}
```

ユーザー向け文字列は `strings/ja.ts` に集約する。内部 stack や未加工のファイル内容を画面へ出さない。
