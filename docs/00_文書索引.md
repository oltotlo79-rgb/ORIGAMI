## 1. 現在地

ユーザー回答で必要な製品判断は確定済みであり、現時点で追加質問はない。未確定なのは、アルゴリズム、SupportProfile、ToleranceProfile、経路証跡形式、入力上限、履歴byte budget、基準機の具体値など、M0Fで試作・測定して固定する技術値である。

## 3. 固定済みの重要判断

- 無料の公開静的Web/PWA、Windows 10/11のChrome・Edge・Firefox。install UI非提供ブラウザは公開URLを正式経路とする。
- ツリー法と本来の格子型箱ひだを別生成pipelineにする。箱ひだは単なる格子snapではない。
- 葉2～20、内部枝幅、長方形、Nx/Ny、枝長・幅の格子近似1%を扱う。
- SupportProfile内では、キャンセルされない限り`verified`または`no-solution-certified`へ終了する。
- 検証済みには、平面から目標までの連続経路、剛体/hinge拘束、全区間CCD、合法接触、動的/終端積層、目標一致の証跡が必要。
- FOLD 1.1/1.2の厳密に1 frameの2D平面CPを読込み、FOLD 1.2を既定出力とする。import直後は常に未検証。
- `.fold`由来projectも表示、M/V編集、保存、再出力、平面/未検証候補3D、対応範囲内での経路検証を許可する。
- 固定timeoutは設けない。3秒/30秒は配置の努力目標で、解なし判定には使わない。
- runtime外部通信、telemetry、広告、外部font/CDN、GPL系依存・参照移植を禁止する。

## 4. M0の実装結果

M0の実装範囲は次のとおり。

1. Git/Vite/React/TypeScript strict、lint、format、Vitest、Playwright、Pages用build。
2. 許可SPDX allowlistと外部origin通信検査。

M0F-0Bでは方式別SupportProfile候補、生成/FOLD入力、CreaseMesh、Target、候補経路、接触、積層の閉じた契約を追加した。これらは`candidate`かつ`scientificClaim: false`であり、runtime parserの合格を検証済み表示へ使えない。

M0Fの最終`npm run m0f:gate`は、単にfixture IDを埋めただけで合格しないようfail-closedである。ID群だけの完備性は`npm run m0f:catalog-gate`で別に確認し、最終gateはreference verifier、mutation、測定済みprofile、M0F reportまで揃うまで失敗終了する。

public repository作成、remote設定、GitHub Pages有効化はリポジトリ所有者の外部操作である。初回push後、同一CI内でqualityと3 browser E2Eに合格したcommitだけをPagesへdeployする。

- 方式別生成用/FOLD検証用SupportProfileと純粋な`checkSupport`。

これらを推測で製品コードへ固定しない。`05_実現性検証仕様書.md` §12.3の全条件を満たした`GO` ADRが、M1開始条件である。
