# OriDesign（仮称）開発計画 v1.1 — v1リリース実装マイルストーン

更新日: 2026-07-14 / ステータス: M0完了、M0F-0A/0B完了、M0F実験実施中（`GO`未達）

v1.0はM0、M0F、M1～M9とPWA公開で構成する。段折りはPhase 2（M10）、Tauriデスクトップ版はv1公開後の別計画とする。

各マイルストーン（M）は実装・レビュー単位であり、記載した受け入れ条件（AC）を全て満たすまで次の依存Mへ進まない。特にM0FはUI製品実装前のハードゲートである。M0FがNo-Goの場合、対応範囲や保証要件をユーザー合意なしに縮小して先へ進めてはならない。

## 0. リリース契約

v1は次を満たす。

- 公開GitHubリポジトリからGitHub Pagesへ配信する無料の静的Web/PWA。バックエンド、外部API、CDN、外部フォント、アクセス解析、広告、クラッシュ送信を使わない。
- Windows 10/11の最新版Chrome、Edge、Firefoxを対象とし、画面幅1280px以上をサポートする。
- ツリー法と、本来の格子型箱ひだをそれぞれ実装する。箱ひだを「ツリー法の葉を格子配置しただけ」の実装にしない。
- M0Fで定義した対応範囲内（葉は最大20枚、固定20葉ケースを含む）の入力は、検証済みCPまたは確認可能な「解なし」を返す。探索の打切りやタイムアウトを解なしと呼ばない。
- 「検証済み」は、厚さゼロの数値モデル上で、平面から、設計projectでは目的の平坦なuniaxial base、FOLD projectではCP/MVに整合する完全平坦目標まで、面の貫通がない連続経路、許可された接触/重なり、整合した積層順序を確認したことを意味する。形式的な数学証明および実物の紙での可折性は保証対象外。
- 計算に強制タイムアウトを設けず、進捗表示とキャンセルを提供する。箱ひだ3秒、ツリー法30秒は基準端末上の努力目標として計測する。
- 用紙寸法は縦横比で保持し、内部計算は紙の短辺を1とする。PDFは指定用紙へフィットし、原寸型紙とはしない。
- ソースはMIT。初期許可SPDX IDは`MIT`、`0BSD`、`BSD-2-Clause`、`BSD-3-Clause`、`Apache-2.0`、`ISC`に限定し、GPL実装を参照移植しない。

## M0: プロジェクト基盤

### 実装

- Vite + React + TypeScript（strict）+ Zustand + Vitest + Playwright + ESLint/Prettierをセットアップする。
- GitHub Actionsにlint、型検査、unit、E2E smoke、production build、依存ライセンス検査を設定する。
- mainからGitHub Pagesへ静的成果物をデプロイする。リポジトリはpublic、LICENSEはMITとする。
- `src/`の最小ディレクトリ、`strings/ja.ts`、テストfixture規約、決定記録用`docs/decisions/`を作る。
- M0F用のCLI/テストハーネスを用意する。設計・生成・検証の製品UIは空のシェルを超えて作らない。ただし、同梱した候補入力だけを外部通信・永続化・製品状態変更なしで再評価し、`scientificClaim: false`、最終判断未記録、製品実装開始未承認を常時明示する診断専用UIは例外とする。
- 本番ビルドが外部オリジンへ通信しないことを検査する仕組みを置く。

### AC

1. クリーン環境でinstall→lint→typecheck→test→buildが成功し、CIが全てグリーンになる。
2. Pagesで設計操作UIのないシェルが開き、診断専用UIを含むPlaywright smokeがChrome/Edge相当/Firefoxの各エンジンで通る。
3. ライセンス検査が`MIT`、`0BSD`、`BSD-2-Clause`、`BSD-3-Clause`、`Apache-2.0`、`ISC`を許可し、GPL系および未知ライセンスで失敗するテストを持つ。
4. production buildをネットワーク監視下で開き、同一オリジンの静的資産以外へリクエストしない。

## M0F: 可折性保証の実現性ゲート（最優先・製品UI実装禁止）

M0Fは使い捨てデモではなく、採否を判断できるテスト可能な参照実装と仕様を成果物にする。GPLコードを読んで移植せず、論文・公開された数理記述から独立実装し、参照文献と独立実装方針を記録する。

### F1. 本来の箱ひだ

- axial、ridge、hinge、river、必要なaxial+N（整数elevationに対応する補助軸線）または同等のpleat routingを含む箱ひだ専用CP構成法を試作する。具体規則はM0Fで固定する。
- 長方形用紙を`Nx`/`Ny`で別管理し、M0Fの数値許容誤差内で正方形セルを維持する。
- 実数の枝長・枝幅（内部枝を含む）を格子値へ近似し、入力値、格子値、相対誤差を返す。最大誤差1%を合格基準とする。
- 格子上限内でより細かい候補、解なし、対応範囲外を区別できるようにする。

### F2. 整合したFOLD面

- 生成線および面情報のない2D FOLDから、交点分割、許容誤差内の頂点統合、重複辺処理、平面グラフ構築、境界抽出、面列挙、向き統一を行う。
- 非平面、非多様体、閉じない領域、不正参照を対象IDと理由付きで拒否する。
- `vertices_coords`、`edges_vertices`、`edges_assignment`、`faces_vertices`が相互整合することを機械検証する。

### F3. 連続折り経路、CCD、積層順序

- 平面状態から目的の完全平坦状態までの、折り線ごとに時刻・速度が異なる連続経路を生成する。
- フレームサンプリングだけでなく、各運動区間に連続衝突判定（CCD）を適用する。接触と最終状態の重なりは許可し、面の貫通は全時刻で禁止する。
- 接触イベントから面の上下関係を作り、経路全体で矛盾しない積層順序を生成・検査する。
- 100%は数学上の完全平坦状態とし、表示用メッシュは数値安定性のため`100%-ε`で評価できるようにする。
- 距離、角度、頂点統合、接触、CCD、経路終端の許容誤差を別々に定義する。

### F4. 対応範囲と終了判定

- ツリー形状、最大次数、枝順、葉数、長さ/幅の数値範囲、用紙比、方式、格子の生成用対応条件と、ツリーを持たないFOLDの頂点/辺/面、assignment、縮退等の検証用対応条件を、別々の決定的predicateとして定義する。
- 葉数上限は20とし、両方式・各葉数2～20に連続的な入力族を持たせる。固定/holdoutの20葉fixture、両方式の内部枝幅、長方形箱ひだを対応範囲に必ず含める。その他の次数・幅・形状制約はこのゲートで確定する。
- predicateを通った入力は、時間制限ではなくアルゴリズムの終了条件により「検証済みCP」またはM0Fで定義する機械確認可能な「解なし根拠」を返す契約にする。
- predicate外は、該当条件、実値、許容範囲、修正候補を返す。数値不安定、例外、キャンセルを「解なし」または「対応範囲外」に偽装しない。

### F5. 固定する技術値と成果物

M0F完了時に次を版管理し、それ以降の仕様が参照できる状態にする。

- 方式別生成用/FOLD検証用の対応範囲predicateと版番号。
- 数値許容誤差プロファイルと版番号。
- `Nx`/`Ny`上限、格子提案規則、1%誤差の計算式。
- `runtime-limits-v1.json`: `.orid.json`/`.fold`のbytes、頂点/辺/面/文字列、`Nx/Ny`、履歴byte budget、Workerキャンセル応答、入力桁数/最小特徴量/最大比率の上限。
- 基準端末（Windows 11、4コアCPU、16GB RAM、内蔵GPU、1920×1080）の具体的CPU/GPU型番とベンチ手順。
- 箱ひだ3秒/ツリー法30秒の計測値、葉数別の折り経路検証時間、メモリ使用量。時間は強制終了条件にしない。
- 解なし根拠のデータ形式と独立checker。
- Go/No-Go報告、既知制約、採用アルゴリズムのADR。

### 必須fixture

fixture ID と期待分類の唯一の正は`05_実現性検証仕様書.md` §9とする。本計画には重複一覧を持たない。正例、負例、敵対例、FOLD検証例、両方式×葉数2～20の生成試験、holdout、mutation、境界試験を全て含む。複数の正解CPがあり得るため、生成座標そのものだけを正解にせず、FOLD整合性、最終形、経路、非貫通、積層、誤差を検査する。

### AC（全項目がGo条件）

1. 規範的なGo条件は`05_実現性検証仕様書.md` §12.3の全14項目とする。本節は要約であり、代替条件ではない。
2. §9の正例はreference verifierが全証跡を受理し、負例・敵対例はmanifestどおりの分類/reason codeで拒否される。
3. 両方式×各葉数2～20の生成試験760件以上、FOLD検証profile内の摂動族、holdout、20葉変形、mutation、境界試験が全件合格する。GEN-TM-20／GEN-BP-20の除外や試験機能扱いは不可。
4. 貫通を意図的に挿入した負例、順序反転、積層循環を検出し、合法な接触/重なりを誤拒否しない。
5. `faces_vertices`欠落FOLDを再構築でき、対応FOLDの完全平坦経路を検証し、非多様体/非平面等を理由付きで拒否する。
6. 方式別/FOLD検証用predicate、解なしchecker、ToleranceProfile、path certificate、canonicalization、runtime limits、基準端末が文書とschema/testで凍結される。
7. 同一入力・seed・profile・検証器版で判定が再現し、キャンセルと計算失敗が解なしにならない。
8. ライセンス/由来監査が完了し、GoのADRを承認する。1項目でも未達ならNo-Goとし、M1以降を開始しない。

## M1: データモデル、スキーマ、永続化

### 実装

- `DesignProject`と、ツリーを持たずsource spec/hash・保持可能custom field・正規化済みFOLD 1.2を持つ`FoldProject`の判別可能unionを定義する。
- UUID、format version、`design`/`layout`/`cp`のrevision vector、各成果物が依存する成分と生成元hash、検証エンジン/対応範囲/許容誤差の版を保存する。
- ツリーに枝ラベル、内部枝を含む幅、明示的な巡回順、`mirrorOf`、軸上自己対称を定義する。箱ひだ格子は`Nx`/`Ny`を持つ。
- CP、安定折り線ID、M/V override、面、TargetState、PathCertificate、区間上界、CCD証跡、接触イベント、動的/終端積層順序、VerificationRecord、判別可能な解なし根拠を型と実行時schemaにする。
- `.orid.json`/`.fold`の実行時検証、migration、重複ID/非有限数/不正参照/上限超過のエラーを実装する。
- Zustand + immer patchによるUndo/Redoを実装する。既定100件、端末設定で変更可能、最大500件。ドラッグと結果採用を各1操作にまとめる。大きなCP/経路はcontent hashで派生blob storeへ置き、履歴は参照を保持する。M0Fで固定するbyte budgetを超えたら未参照の古い履歴から破棄する。
- File System Access APIの上書き保存とダウンロードfallbackを実装する。上書きAPI成功またはdownload後の明示確認でのみdirtyを解除し、開始だけでは解除しない。
- 編集停止2秒後に、仮レコードからcurrent pointerを同一transactionで切り替える原子的IndexedDBスナップショットを実装する。最近一覧は更新日時降順10件とし、open/lock中を保護して最古から決定的にevictする。復元/破棄、個別削除を実装する。
- BroadcastChannelとheartbeatでUUID単位の複数タブロック、読み取り専用、明示的編集引継ぎを実装する。heartbeat消失時も確認を挟み、競合snapshotは比較選択させて自動mergeしない。
- CP色、3D色、履歴上限等を端末共通設定としてプロジェクト外に保存する。

### AC

1. 両project種別について保存→再読込が意味的に完全一致し、verified成果物の全evidence hashが同梱blobへ解決する。欠落blobを含む全schemaエラーがJSONパスと理由を返す。
2. migration、重複ID、非有限数、不正参照、M0F上限超過、検証版不一致のunit testが通る。
3. 編集停止2秒後に原子的snapshotが作られ、途中書込みでも直前の完全版を復元できる。起動時は自動復元せず復元/破棄を選べ、保存後も復旧データが残り、11件目の決定的evictionが仕様どおりになるE2Eが通る。
4. 同一UUIDを2タブで開くと後発が読み取り専用になり、明示引継ぎ、heartbeat消失、競合snapshot選択で無言の上書きが起きないE2Eが通る。
5. Undo/Redoが既定100件、最大500件とM0F byte budgetの双方に従い、ドラッグ1回と結果採用1回がそれぞれ1履歴になる。参照中blobはevictされない。
6. Chrome/Edge系では上書き、Firefox fallbackではダウンロード確認フローとなる。cancel/permission denial/開始のみ/「未保存のまま」でdirtyが解除されず、上書き成功または利用者確認後だけ解除される。

## M2: アプリシェルとツリーエディタ

### 実装

- 画面仕様書§1～§4、§8、§12の共通ヘッダー、3ペイン、設計/FOLDタブ構成、プロパティ、ツリーエディタを実装する。
- ノード/枝作成、削除、ラベル/長さ/内部枝幅編集、キャンバスと表の同期、巡回順の保存/手動調整を実装する。
- 対称ONで実ノード/枝を`mirrorOf`対応させ、軸上要素を自己対称にする。OFFで左右両方を保持する。
- 不正な編集中状態を許可しつつ、連結、閉路、葉数、値、巡回順、M0F対応範囲の常時検証と生成disabled理由を表示する。
- 方式変更時は確認後、ツリーだけを維持し、配置以降を破棄する。箱ひだ切替時は格子提案への導線を出す。
- ツリーの全主要操作にキーボード代替と読み上げ可能なラベルを付ける。

### AC

1. E2E「ノード追加→枝接続→ラベル/長さ/内部枝幅編集→巡回順変更→Undo/Redo」がキャンバスと表で一致する。
2. 対称ONで対応要素が生成され、軸上要素は自己対称、OFF後は左右要素が独立して残る。
3. 未接続、閉路、葉不足、非正長、葉21、曖昧な巡回順の各fixtureで正しい理由が出て、生成が無効になる。
4. マウスを使わずにツリー作成、要素選択、数値編集、削除ができるPlaywright E2Eが通る。
5. 1280px未満で警告が表示され、1280px以上では3ペインの主要操作が隠れない。

## M3: CP/FOLD、面構築、エクスポート

### 実装

- M0Fの面構築参照実装を製品コードへ移し、FOLD 1.1/1.2の正式対応サブセットを読込・検証し、FOLD 1.2を既定出力にする。
- `file_frames`が不存在または空のtop-level key frame 2D CPを受付け、`faces_vertices`欠落時に再構築する。複数フレーム、初期3D、非平面、非多様体、assignment `C/J`は理由付きで拒否する。
- FOLDプロジェクトでsource spec/hash、保持可能custom field、正規化済みFOLD 1.2、CP表示、平面/未検証候補3D、M/V編集、目標/PathCertificate、`.orid.json`保存、再出力を扱う。
- CPビューのズーム、パン、線種/記号、格子、座標、折り線一覧、安定IDに対するM/V override、局所検証overlayを実装する。
- FOLD/SVG/PNG/PDF exporterを実装する。PNG既定4096px/最大16384px、PDF 1ページ/10mm余白/縦横自動/中央フィット、SVG/FOLD正規化座標とする。
- 未検証/stale出力に確認、検証状態メタデータ、必要な可視フッターを付ける。

### AC

1. `FOLD-001`/`GEO-002`/`GEO-003`: `faces_vertices`あり/なしの既知FOLDが同じ面トポロジになり、非対応fixtureは対象と理由を表示する。
2. FOLD読込直後が`unverified`で、CP表示→M/V編集→`.orid.json`保存→再読込→4形式再出力が通る。
3. `FOLD-002`/`FOLD-003`: parallel arrayとcustom field保持/除外規則を検査し、出力FOLD 1.2を独立した2つの対応toolで開ける。
4. `EXPORT-001`～`EXPORT-003`: FOLD/SVGの正規化座標、PNGの4096/16384境界と透明背景、PDFの余白/向き/中央フィットを自動検査する。
5. 未検証/staleを同意なしで出力できず、同意後の全形式で状態を確認できる。
6. CP上の折り線をキーボードで選択し、M/V変更と局所検証結果の確認ができる。

## M4: 折り経路検証エンジン、CCD、積層、3D

### 実装

- M0Fで採用した折り経路生成、連続自己衝突判定、接触分類、動的/終端積層順序、終端検査を製品モジュールへ実装する。planner/generatorと独立したreferenceVerifierを別モジュールに保つ。
- 設計projectはuniaxial tree base、FOLD projectはM/Vを尊重しUを明示解決する完全平坦状態をTargetStateとし、workflow別SupportProfileを適用する。
- 全工程をWorkerで実行し、job ID、対象revision vector、意味的入力hash、工程別進捗、キャンセルを扱う。固定タイムアウトを設けない。
- 3Dを検証済みPathCertificateの時刻`t=0..100%`で再生し、先頭/末尾、再生、一時停止、逆再生、任意seekを実装する。100%表示は`100%-ε`の平坦直前メッシュとする。
- 未検証FOLDは平面表示を許可する。候補経路がある場合は、警告付きの別操作として未検証候補を表示できるが、検証済みと同じ外観・既定再生を使わない。検証成功後のみ検証済み経路アニメーションを有効にする。
- 距離・角度・CCD許容誤差、接触、最小離隔、積層順序をUIへ表示する。
- M/V編集後に経路と検証を無効化し、再検証成功まで戻さない。

### AC

1. `VERIFY-002`～`VERIFY-005`/`FOLD-004`: M0Fの全経路fixtureが製品エンジンでも同じ合否となり、TargetState、全区間上界、CCD証跡本体、接触イベント、動的/終端積層を保存して独立再検証できる。
2. 面の接触/最終重なり正例は合格し、途中貫通、積層循環、終端未達の負例は該当時刻/面ID付きで失敗する。
3. 3Dで折り線ごとに異なる時系列が再生され、先頭/末尾、再生、一時停止、逆再生、任意seekがPathCertificateと一致し、100%ラベルと表示用`ε`が検証プロファイルに一致する。
4. 1000三角形で30fps以上をM0F基準端末で満たす。2000三角形の値も記録するがリリース阻害条件にはしない。
5. 実行中編集、キャンセル、古いjob完了の全ケースで最後の検証済み結果が上書きされないE2Eが通る。

## M5: ソルバー共通基盤とジョブ統合

### 実装

- M0Fで採用・図示した長さ/内部枝幅の測定量、全葉ペアのパス条件、用紙内包含、アクティブグラフ/領域抽出、幾何predicateを実装する。幅を単純に木距離へ加算するとは仮定しない。
- 保存済み巡回順を用いた平面埋込みと、生成線からM3の面構築へ渡す共通インターフェースを実装する。
- 前川・川崎検査を実装し、川崎違反をM/V編集で修正可能とは表示しない。
- 安定した生成折り線IDとM/V割当て候補を作り、前川違反は手動編集可能な下書き、川崎/トポロジ違反は未検証下書きまたは生成失敗として返す。
- 配置→CP→面→経路→CCD→積層→判定を1つのジョブpipelineとして接続する。各段階の結果は必要なrevision vector成分と意味的入力hashが一致するときだけcommitする。
- `verified`、`no-solution-certified`、`cancelled`、`unsupported`、`invalid`、`internal-error`を混同しない`SolveResult`と、`none`/`stale`/`unverified`/`verifying`を含む画面状態の写像を実装する。

### AC

1. パス条件が幅0/葉枝幅/内部枝幅を含む既知例の理論値と一致し、全葉ペア違反を対象IDと量付きで返す。
2. rabbit ear、waterbombの共通幾何と前川/川崎検査がM0F許容誤差内で期待不変条件を満たす。
3. 川崎違反のM/V変更では合格にならず、前川違反はM/V変更後に再検証が要求される。
4. pipelineの各段階で注入した失敗、キャンセル、revision競合が既存結果を破壊せず、解なしへ誤分類されない。
5. `QUALITY-001`: model/geometry/solver/planner/verify/referenceVerifierのunit testカバレッジ80%以上を達成する。

## M6: 本来の箱ひだソルバー

### 実装

- M0Fで採用した箱ひだ専用構成法を製品化し、axial、river、ridge、hinge、必要なaxial+Nまたは同等のpleat routingを生成する。
- 長方形の`Nx`/`Ny`、正方形セル検査、自動提案上位3件、手動指定、格子点配置を実装する。
- 長さと内部枝を含む幅を格子近似し、入力値/格子値/相対誤差を表示する。1%超過時は細分候補、解なし、対応範囲外を区別する。
- 自動配置と格子スナップ手動配置、違反ペア/量、アクティブパス、進捗、キャンセルを配置UIへ接続する。
- 箱ひだ生成結果をM4へ渡し、検証済み経路または確認可能な解なしまで実行する。

### AC

1. GEN-BP-RECTで正方形セル、`Nx`/`Ny`、全長さ/幅の誤差1%以下、専用箱ひだ線種、FOLD整合性を満たす。
2. GEN-BP-BIRD-4、GEN-BP-INSECT-8、GEN-BP-WIDTH-INTERNAL、GEN-BP-20が配置→CP→面→経路→CCD→積層まで合格し、箱ひだでも葉20を正式な保証対象に含める。
3. 格子提案はM0Fの上限内から合格候補だけを順位付きで返し、1%超過、上限超過、解なしを正しく分類する。
4. GEN-BP-GRID-4で`Nx=4`または`Ny=4`を受け付け、長方形で単一`N`に丸める回帰を防ぐ。
5. M0F基準端末で箱ひだ配置時間を測定し、3秒努力目標の成否を表示する。3秒超過でもジョブを強制終了しない。

## M7: ツリー法ソルバーと葉20保証

### 実装

- 保存済み巡回順を持つツリー埋込み、連続配置最適化、対称制約、手動微調整を実装する。
- M0Fで採用したツリー法CP構成を実装し、幅付き内部枝と分岐部のriverを扱う。収まらない場合は確認可能な解なしとする。
- M0F対応範囲predicateを生成前に適用し、対応範囲内の入力は検証済みCPまたは解なし根拠まで計算する。
- 葉20までの計算をWorkerで実行し、現在の最良スケール、工程、経過時間、キャンセルを表示する。30秒は努力目標でありタイムアウトにしない。
- 配置からM4の検証済み3Dまで一気通貫にする。

### AC

1. GEN-TM-BIRD-4、GEN-TM-INSECT-8、GEN-TM-WIDTH-INTERNAL、GEN-TM-20が全pipelineを通り、M0F独立checkerにも合格する。REF-RABBIT-EARとREF-WATERBOMBの共通検証はM5 ACで扱う。
2. GEN-TM-20は試験機能や警告付き対象ではなく正式な保証対象としてUIに「検証済み」を表示する。
3. 対応範囲内の既知解なしfixtureは根拠checkerに合格し、対応範囲外fixtureは条件/実値/修正候補を返す。
4. 対称ONの配置とCPが対応関係を維持し、OFF後の左右要素が独立しても巡回順を失わない。
5. M0F基準端末で葉数別時間を記録し、葉20の30秒努力目標の成否を表示する。30秒超過を解なしにしない。

## 横断ルール

1. 各MはCIグリーン、レビュー、関連文書更新、Pagesデモ確認後に完了とする。常にmainをデプロイ可能に保つ。
2. ACのうち機械判定できるものは必ず自動化する。目視項目は補助に留め、PRへスクリーンショット/動画と環境情報を添付する。
3. solverの乱数はseedを注入可能にし、fixtureはseed、許容誤差profile、エンジン版を固定する。
4. 結果commitはjob ID、必要なrevision vector成分、意味的入力hashの一致を必須とし、途中結果、キャンセル、古いWorker応答で既存データを上書きしない。
5. M0Fで固定した対応範囲、許容誤差、上限、解なし根拠形式を変更する場合はADR、全fixture再実行、保証レビューを必須とする。
6. GPL実装は依存導入だけでなくコード参照移植も禁止する。数理記述から独立実装し、参照文献と設計判断をADRに残す。
7. 箱ひだ3秒、ツリー法30秒、2000三角形30fpsは努力目標。1000三角形30fpsと全必須fixtureの数値保証は必須条件である。
8. 未検証CPの表示・編集・出力は許可するが、検証済みと同じ表示、既定の経路再生、無警告出力を許可しない。
